# 16.3 安全团队职能划分

## 概述

组织架构选型完成后，下一步需要明确各职能的边界与协作机制。职能划分不清是安全体系失效的常见原因——多个团队各自"尽职"，但关键场景无人负总责。

某互联网公司曾发生一起数据泄露事件，客户手机号和订单信息被爬虫持续抓取三个月才被发现。事后复盘显示，应用安全团队认为 API 限流不属于代码扫描范畴，SOC 团队监控到异常流量但不清楚业务逻辑，无法判断是否为攻击，基础设施团队则认为业务层风控应由开发团队负责。三个团队的职责定义均未覆盖「业务 API 被恶意调用」这一场景，形成了典型的「三不管」地带。

本节定义十大核心安全职能，明确职责边界、协作机制与人员配置参考，解决「谁负责什么」的问题。

职能划分的核心原则是"无缝覆盖"——每一个安全场景都应该有明确的责任人，既不能有空白（无人负责），也不能有过度重叠（多人负责但无人负总责）。好的职能划分应该让任何一个安全问题都能快速找到对应的团队，而不是在多个团队之间踢皮球。

---

## 16.3.1 十大核心安全职能

### 职能体系结构

安全组织的职能可划分为四个层次。这种分层设计借鉴了军事组织的指挥层级思想：战略层负责"打什么仗"，架构层负责"怎么打"，技术层负责"用什么武器"，运营层负责"日常巡逻和应急响应"。每一层都不可或缺，缺少战略层会导致安全工作缺乏方向感，缺少运营层则无法发现和应对实际威胁。

**战略与治理层**

- 安全战略与治理
- GRC 与合规

**架构与设计层**

- 安全架构与工程
- 产品安全

**技术防护层**

- 应用安全
- 基础设施安全
- 云安全
- 数据安全与隐私

**运营与响应层**

- 安全运营（SOC）
- 红队与攻防

此外，安全工程与自动化作为横向支撑职能，为上述各层提供工具开发和流程自动化能力。

## 16.3.2 职能详细定义

### 1. 安全战略与治理

安全战略与治理职能负责将安全工作与企业战略对齐，是安全组织与高管层、业务部门的主要接口。

#### 核心职责

**战略规划**

制定年度安全战略需要对齐公司业务规划，将「保护公司资产」这一抽象目标转化为可衡量的目标。若公司计划进入新市场，安全战略的优先级可能从内部监控转向合规认证。编制安全预算时，核心约束通常是「业务增长幅度与安全预算增幅不匹配」，需要通过自动化、集中采购或安全左移等方式在资源受限条件下维持安全水平。

**治理框架**

建立安全决策机制需要在集中管控与业务灵活性之间找平衡点。常规变更可通过预定义规则自动审批，高风险变更（如开放公网权限、生产数据导出）需要委员会评审。定义安全原则和红线时，需配套验证机制和例外流程。

**高管沟通**

董事会汇报的核心挑战是向非技术背景的决策者说明安全投入的价值。技术团队习惯用「阻止攻击次数」量化价值，但决策层更关心「避免的潜在损失」。与 CEO/CTO 同步时，应按「紧急且需要决策的事项」优先级排序。危机事件对外沟通时，需在透明度与避免恐慌之间平衡。

**业务对齐**

BISO 机制的运作需要持续投入。每周与各业务线 BISO 的同步可提前识别安全需求，使安全团队有时间设计风险控制方案，而非被动应对。参与业务规划会议可避免「产品设计定稿后才发现架构有安全缺陷」的情况。

#### 关键约束

- **可衡量性约束**：抽象目标难以量化，需转化为可测量指标
- **决策速度约束**：紧急变更不能等待周期性会议，需设计快速决策通道
- **预算周期约束**：年度预算与突发安全事件的时间错配，需预留应急预算
- **人员能力约束**：技术理解力与业务影响力较难兼具，通常需要团队互补

#### 团队配置参考

**初创期（安全团队 3-5 人）**：CISO 身兼数职，配 1 个战略助理或兼职 BISO。

**成长期（安全团队 10-30 人）**：CISO 专注战略和沟通，配置安全战略经理、2-3 个 BISO、安全文化推广专员。

**成熟期（安全团队 50+ 人）**：完整团队包括 CSO/VP、2-3 个 Director 级别、6-8 个 BISO、项目管理办公室。

#### 常见误区

1. **战略与执行脱节**：制定的安全战略过于理想化，未考虑现有人员能力和预算限制，导致无法落地
2. **沟通语言错位**：向高管汇报时使用技术语言，无法获得业务认同和资源支持
3. **BISO 机制流于形式**：BISO 只是名义上存在，缺少实际的信息同步和协调机制

#### 验证方法

- 通过季度调研评估业务部门对安全支持的满意度
- 跟踪安全项目按时交付率
- 评估安全预算与资源获取情况

#### 运行指标

- 安全战略与业务战略对齐度（业务负责人认可比例）
- 安全项目按时交付率
- 高管沟通频率与反馈

### 2. GRC 与合规

GRC 职能涵盖治理（Governance）、风险（Risk）、合规（Compliance）三个维度，负责建立和维护安全管理体系。

GRC 团队常被误解为"写文档的团队"，但其真正价值在于将抽象的安全要求转化为可验证的控制措施。一位 GRC 经理曾这样描述他的工作："我们是安全团队的翻译官，把监管语言翻译成技术要求，把技术状态翻译成合规证据。"没有 GRC 职能的安全组织，往往在面对外部审计或客户安全评估时陷入被动——技术能力再强，如果无法用合规语言证明，也难以获得客户信任。

#### 核心职责

**治理**

- 制定安全政策体系（通常为顶层政策、标准、操作指南三层结构）
- 维护安全控制框架
- 运作安全委员会

**风险管理**

- 风险评估（年度全面评估配合季度重点评估）
- 风险登记册管理
- 第三方风险管理
- 供应链风险评估

**合规认证**

- ISO 27001、SOC 2、PCI DSS 等认证的准备和维护
- GDPR、CCPA、PIPL 等隐私法规的合规实施
- 审计响应
- 合规差距分析

#### 适用边界

GRC 职能在以下场景价值突出：

- 有外部合规要求（客户要求、监管要求、上市要求）
- 需要系统化管理安全风险
- 组织规模达到需要正式政策体系的程度

对于早期初创公司，GRC 工作可由 CISO 或外部顾问兼任。

#### 关键约束

- **审计周期约束**：SOC 2 Type II 需要至少 6 个月的控制措施运行证明
- **证据收集成本**：每次审计需要准备大量证据文件
- **政策执行落差**：政策制定容易，执行和验证困难

#### 团队配置参考

**初创期**：1-2 人或外包给咨询公司。

**成长期**：GRC Manager、2 个 Compliance Specialist、1 个 Risk Analyst。

**成熟期**：GRC Director、按地域或产品线分工的 Compliance Manager、专项 Compliance Analyst、Risk Analyst 团队、Policy Manager。

#### 常见误区

1. **合规等于安全**：通过合规认证不代表安全能力到位，合规是最低要求而非目标
2. **政策与实践脱节**：政策文档齐全但实际执行情况未验证
3. **风险评估流于形式**：年度风险评估只是走流程，未真正影响资源分配

#### 验证方法

- 内部审计验证控制措施执行情况
- 抽查政策执行的证据
- 跟踪风险登记册中风险处置进度

#### 运行指标

- 合规认证覆盖率
- 审计发现项关闭率及时效
- 政策更新频率
- 第三方风险评估覆盖率

### 3. 安全架构与工程

安全架构职能负责设计企业级安全架构、进行威胁建模、制定技术标准。

安全架构师的角色类似于城市规划师——不亲自建造每一栋楼，但决定城市的整体布局、交通动线和公共设施位置。好的安全架构能让后续的安全实施工作事半功倍，而架构缺陷则会导致安全债务持续累积。某电商公司的案例很有代表性：早期为了快速上线，认证系统采用了"各应用独立管理用户"的设计，导致后来整合统一身份时花费了两年时间和数百万成本。如果当初有架构师参与初始设计，这笔技术债务完全可以避免。

#### 核心职责

**安全架构设计**

- 企业级安全架构蓝图
- 领域架构（身份认证、数据安全、云安全、网络安全）
- 零信任架构规划

**威胁建模**

- 新系统上线前的威胁分析（STRIDE、PASTA 等方法）
- 识别攻击路径和缓解措施
- 输出威胁模型文档

**安全设计评审**

- 审查重大项目的安全设计
- 提出架构改进建议
- 跟踪安全控制措施落地

**技术选型与标准**

- 制定技术白名单
- 评估新技术的安全风险
- 制定安全基线

**架构债务管理**

- 识别技术欠债
- 制定偿还计划
- 平衡业务需求和架构优化

#### 适用边界

安全架构职能在以下场景必要：

- 系统复杂度高，涉及多个技术栈和数据流
- 有重大技术变更（云迁移、微服务改造）
- 需要制定统一的安全技术标准

对于技术栈单一、系统简单的组织，架构工作可由资深工程师兼任。

#### 关键约束

- **架构评审周期与业务速度的矛盾**：评审过慢会阻塞业务，过快则可能遗漏风险
- **架构债务持续累积**：业务压力下往往优先交付功能，架构债务难以偿还
- **跨团队协调成本**：架构决策需要多个团队配合执行

#### 团队配置参考

**初创期**：1 个架构师身兼数职，重大项目请外部咨询。

**成长期**：Principal Architect、2-3 个领域架构师。

**成熟期**：Chief Architect（VP 级别）、5-8 个领域专家、2-3 个威胁建模专家。

#### 常见误区

1. **架构设计脱离实现**：设计方案过于理想化，未考虑实现成本和现有系统约束
2. **威胁建模覆盖不足**：只对新系统做威胁建模，存量系统风险未评估
3. **技术标准更新滞后**：标准制定后未随技术演进更新，失去指导价值

#### 验证方法

- 跟踪重大项目安全评审覆盖率
- 验证威胁模型中识别的风险是否有对应控制措施
- 评估架构债务偿还进度

#### 运行指标

- 重大项目安全评审覆盖率
- 威胁建模覆盖率（高风险项目）
- 架构债务偿还率

### 4. 产品安全

产品安全职能仅适用于将软件作为产品销售的公司（B2B SaaS、软件产品），负责产品的安全生命周期管理和客户安全交付。

产品安全与应用安全的本质区别在于"客户视角"。应用安全关注的是代码有没有漏洞，产品安全关注的是客户能否信任这个产品。一个典型场景：客户在采购评估时发来 500 题的安全问卷，要求在一周内完成；客户的安全团队要求提供 SOC 2 报告和渗透测试结果；签约后客户还会定期进行供应商安全评估。这些工作都需要专人负责协调，既懂技术能回答深度问题，又懂销售流程知道如何与客户沟通。产品安全团队就是这个"客户安全界面"的维护者。

#### 核心职责

**SDL 流程管理**

- 制定并推广安全开发生命周期最佳实践
- 在产品生命周期各阶段嵌入安全检查点
- 培训研发团队理解安全需求

**漏洞生命周期管理**

- 接收外部漏洞报告
- 漏洞分级、修复跟踪
- CVE 申请与披露
- 补丁发布和客户通知

**客户安全交付**

- 回答客户安全问卷
- 提供安全文档（架构图、数据流、渗透测试报告）
- 支持客户的第三方审计

**产品安全特性开发**

- 定义产品需要的安全特性（SSO、审计日志、RBAC）
- 与产品经理协作排优先级
- 审查实现方案

#### 适用边界

产品安全职能适用于：

- B2B SaaS 公司
- 软件产品公司
- 需要对外提供安全承诺的产品

内部系统为主的公司不需要独立的产品安全团队。

#### 关键约束

- **客户安全问卷响应成本**：大客户的安全问卷可能有数百个问题
- **漏洞披露时机决策**：过早披露可能被利用，过晚披露影响客户信任
- **安全特性与产品功能的优先级竞争**

#### 团队配置参考

**初创期（种子轮 - A 轮）**：1-2 人，由 AppSec 兼任。

**成长期（B 轮 - C 轮）**：Product Security Manager、2-3 个 Security Engineer。

**成熟期（上市公司）**：Director of Product Security、产品安全工程师团队、客户安全顾问、漏洞协调员。

#### 常见误区

1. **漏洞响应流程缺失**：接到漏洞报告后无明确的分级和修复流程
2. **安全文档与实际不符**：提供给客户的安全白皮书与实际实现有差距
3. **CVE 管理混乱**：未建立规范的 CVE 申请和披露流程

#### 验证方法

- 模拟客户安全评估流程
- 验证漏洞响应 SLA 执行情况
- 审查安全文档准确性

#### 运行指标

- 客户安全问卷响应时效
- 漏洞修复 SLA 达成率
- CVE 披露及时性

### 5. 应用安全

应用安全职能负责保障应用程序的安全性，是与研发团队协作最紧密的安全职能。

应用安全团队的工作可以用"质检员"和"教练"两个角色来理解。作为质检员，AppSec 需要发现代码中的安全缺陷；作为教练，AppSec 需要教会开发团队写出安全的代码。前者是短期目标，后者才是长期价值。一个成熟的 AppSec 团队应该追求"教会开发团队不需要我们"的状态，而不是让自己成为所有安全问题的瓶颈。这就是为什么 Security Champions 项目如此重要——通过在每个开发团队培养一个安全代言人，AppSec 的影响力可以成倍放大。

#### 核心职责

**安全测试工具运营**

运营 SAST/DAST/IAST/SCA 工具时，核心挑战是「扫描覆盖率」与「误报率」的矛盾。启用所有检查规则可发现更多潜在问题，但误报率可能导致开发团队失去信任。实践中需要分阶段调优，先收集真实漏洞样本建立「已知误报库」，再基于历史数据调整规则权重。

工具选型需权衡「扫描速度」与「检测深度」。对于 CI/CD 流水线，速度是硬约束，通常采用「轻量级工具在每次提交时运行 + 深度工具在夜间批量扫描」的组合策略。

**代码安全审查与咨询**

人工代码审查成本高，需选择性投入。判断标准包括，高价值模块（认证/授权逻辑、支付流程、敏感数据处理）、高风险变更（涉及权限绕过的代码修改）、新技术栈（团队第一次使用新框架）。

**DevSecOps 集成与自动化**

在 CI/CD 流水线中集成安全工具时，核心约束是不能显著延长构建时间。解决方案包括并行执行、增量扫描、智能跳过（只改配置文件时跳过代码扫描）。

设计安全门禁策略时需在一致性与灵活性之间权衡。成熟的门禁策略是「风险分级 + 临时豁免机制」，Critical 级别（认证绕过、SQL 注入、RCE）必须修复无豁免，High 级别可申请临时豁免但需限期修复，Medium/Low 级别记录到 backlog 排期。

**漏洞全生命周期管理**

漏洞优先级排序不能仅依赖 CVSS 评分，需多维度评估，技术严重性、暴露面（公网可达程度）、资产价值、可利用性（是否有公开 EXP）。

修复验证需包括代码 Review、回归测试（用原始 PoC 重新测试）、旁路测试（尝试绕过修复方案）。

#### 适用边界

应用安全职能在有自研应用的组织中必要。对于完全使用 SaaS 产品的组织，应用安全工作主要体现在供应商安全评估中。

#### 关键约束

- **开发团队容忍度约束**：误报率过高会导致开发团队忽略所有报告
- **修复资源约束**：研发团队用于修复安全问题的时间有限，推送的漏洞工作量需控制在可消化范围
- **工具成本约束**：商业 SAST 工具授权费用与研发团队规模相关
- **技能传递约束**：AppSec 团队的核心价值是「教会开发团队写安全代码」，需要长期投入

#### 团队配置参考

人员比例参考：

- 金融、支付类，每 50 研发配 1 AppSec
- 一般互联网，每 100 研发配 1 AppSec
- 内部系统为主，每 150 研发配 1 AppSec

**初创期（研发 10-50 人）**：1-2 个 AppSec。

**成长期（研发 100-300 人）**：AppSec Manager、4-6 个 AppSec Engineer。

**成熟期（研发 500+ 人）**：AppSec Director、安全测试组、DevSecOps 组、代码审查组、security champions 协调组。

#### 常见误区

1. **只做扫描不做赋能**：AppSec 团队只输出漏洞报告，不帮助开发团队理解如何修复
2. **安全门禁过于僵化**：所有高危漏洞一刀切必须修复，不考虑业务紧急程度和实际风险
3. **工具依赖过重**：完全依赖自动化工具，忽视人工审查的价值

#### 验证方法

- 抽查漏洞修复质量（是否真正修复而非表面修复）
- 跟踪 security champions 培养效果
- 评估开发团队对 AppSec 服务的满意度

#### 运行指标

- 应用安全测试覆盖率
- 高危漏洞修复 SLA 达成率
- 误报率
- 漏洞引入率（每千行代码新增漏洞数）

### 6. 基础设施安全

基础设施安全职能负责网络、终端、身份认证等基础设施层面的安全。随着云化推进，该职能与云安全的边界需要明确划分。

基础设施安全是安全体系的"地基"。无论应用层安全做得多好，如果攻击者能通过网络漏洞直接进入内网，或者通过终端恶意软件窃取凭据，上层的安全措施都会被绕过。基础设施安全团队需要与 IT 团队紧密协作，但职责分工需要明确：安全团队负责制定策略和标准（例如"所有特权账号必须启用 MFA"），IT 团队负责执行落地（例如配置 Azure AD 条件访问策略）。这种"安全制定政策，IT 执行操作"的分工模式是跨职能协作的典型范例。

#### 核心职责

**网络安全**

- 防火墙、IDS/IPS 策略管理
- 网络隔离（DMZ、内网分区、微隔离）
- DDoS 防护
- VPN 与远程访问

**终端安全**

- EDR 部署和运营
- 反病毒/反恶意软件
- 终端加密
- BYOD/MDM（移动设备管理）

**身份与访问管理（IAM）**

- SSO（单点登录）
- MFA（多因素认证）
- PAM（特权访问管理）
- 身份治理（定期 Access Review）

**系统加固**

- 安全基线配置（CIS Benchmark）
- 补丁管理
- 漏洞扫描

#### 适用边界

基础设施安全在有自建基础设施的组织中必要。对于完全云化的组织，该职能可与云安全合并。

#### 关键约束

- **与 IT 团队的职责划分**：需明确安全团队制定策略、IT 团队执行运维的分工
- **终端多样性**：Windows、macOS、Linux、移动设备需要不同的安全方案
- **遗留系统兼容性**：老旧系统可能不支持现代安全措施（如 MFA）

#### 团队配置参考

**初创期**：1-2 人，通常与 IT 团队合并。

**成长期**：Infrastructure Security Manager、网络安全工程师、终端安全工程师、IAM 工程师。

**成熟期**：Director、网络安全团队、终端安全团队、IAM 团队、系统加固专家。

#### 常见误区

1. **防火墙规则失控**：规则长期累积未清理，存在冲突和失效规则
2. **补丁管理滞后**：高危补丁未及时部署
3. **特权账号管理松散**：特权账号过多且缺乏监控

#### 验证方法

- 防火墙规则审计
- 补丁部署状态检查
- 特权账号清单审查
- EDR 覆盖率检查

#### 运行指标

- 终端 EDR 覆盖率
- 补丁及时率
- MFA 覆盖率（特权账号、普通账号）
- 防火墙规则审计频率

### 7. 云安全

云安全职能负责云环境的安全架构、配置合规和运营。与基础设施安全的区别在于云环境特有的安全挑战（配置错误、权限过大、多云管理）。

云安全面临的核心挑战是"速度"。传统数据中心环境中，新服务器上线需要经过采购、安装、配置等流程，安全团队有充足时间进行安全评审。但在云环境中，开发人员可以在几分钟内创建几十个云资源，安全团队根本来不及逐一审查。这就是为什么云安全必须依赖自动化——通过 Policy as Code 将安全策略编码，通过 CSPM 持续监控配置偏离，通过 Guardrails 在资源创建时就阻止违规配置。云安全团队的工作重心不是"审批每一个资源"，而是"建立自动化的安全护栏"。

#### 核心职责

**云安全架构**

- 多云安全架构设计
- 云迁移安全评估
- 云安全参考架构（Landing Zone、账号结构、网络设计）

**云安全工具运营**

- CSPM（Cloud Security Posture Management，配置合规）
- CWPP（Cloud Workload Protection Platform，工作负载防护）
- CIEM（Cloud Infrastructure Entitlement Management，权限管理）

**容器与 Kubernetes 安全**

- 容器镜像扫描
- Kubernetes 安全加固
- Service Mesh 安全

**云合规**

- 云配置基线（CIS Benchmark）
- 云审计日志分析
- 自动化合规检查（Policy as Code）

#### 适用边界

云安全职能在以下场景需要独立设置：

- 云资源规模大（数百个以上云资源）
- 多云环境
- 使用容器和 Kubernetes

对于云资源较少的组织，云安全可与基础设施安全合并。

#### 关键约束

- **云配置变更速度快**：开发团队可以快速创建云资源，安全检查需要自动化
- **多云管理复杂性**：不同云平台的安全机制差异大
- **共享责任模型理解**：需明确云厂商与企业的责任边界

#### 团队配置参考

**初创期（云资源少）**：与 InfraSec 合并。

**成长期（多云环境）**：Cloud Security Architect、2-3 个 Cloud Security Engineer、Kubernetes Security Specialist。

**成熟期（多云、多地域）**：Director of Cloud Security、按云平台分工的架构师、云安全工程师团队、容器安全专家。

#### 常见误区

1. **配置检查滞后**：只在上线时检查，运行中的配置变更未监控
2. **权限过大**：云账号权限未遵循最小权限原则
3. **容器镜像未扫描**：直接使用公开镜像未进行安全扫描

#### 验证方法

- CSPM 合规报告
- 云账号权限审计
- 容器镜像扫描覆盖率检查

#### 运行指标

- 云配置合规率（CIS Benchmark）
- 云账号 IAM 最小权限覆盖率
- 容器镜像扫描覆盖率
- 配置异常检测时效

### 8. 数据安全与隐私

数据安全与隐私职能负责数据分类、加密、防泄露以及隐私合规。随着隐私法规趋严，该职能的重要性持续提升。

数据是企业最核心的资产，也是攻击者的终极目标。无论攻击路径多么复杂，最终目的通常都是窃取或破坏数据。数据安全职能需要回答一个根本问题："我们有什么数据，在哪里，谁能访问，如何保护？"听起来简单，但大多数组织无法完整回答这四个问题。数据分类是一切的起点——如果不知道哪些是敏感数据，就无法确定保护优先级。隐私合规则是数据安全的法律约束，GDPR 的天价罚款（最高可达全球年营收的 4%）让数据隐私从"合规成本"变成了"业务风险"。

#### 核心职责

**数据安全**

- 数据分类与标记
- 数据加密（静态加密、传输加密）
- 密钥管理
- 数据丢失防护（DLP）
- 数据访问控制

**隐私合规**

- GDPR、CCPA、PIPL 等法规的合规实施
- 隐私影响评估（DPIA）
- 数据主体权利响应（DSAR）
- Cookie 合规
- 跨境数据传输

**信息保护**

- 信息分类
- IRM（信息权限管理）
- 内部威胁检测

#### 适用边界

数据安全与隐私职能在以下场景需要独立设置：

- 有跨境业务（涉及 GDPR、PIPL 等法规）
- 处理大量敏感个人信息
- 行业有特殊数据保护要求（金融、医疗）

对于数据类型单一、无跨境业务的组织，该职能可与 GRC 合并。

#### 关键约束

- **数据分类覆盖成本**：存量数据量大时，分类标记工作量大
- **DLP 误报率**：DLP 规则过严会影响正常业务
- **DSAR 响应时效**：GDPR 要求 30 天内响应，需要跨系统协调

#### 团队配置参考

**初创期**：由 GRC 兼任。

**成长期（有跨境业务）**：Data Security Manager、DPO/Privacy Officer、数据安全工程师、隐私分析师。

**成熟期（多地域、多业务）**：Director of Data Security & Privacy、数据安全工程师团队、隐私工程师、按地域分工的 DPO、数据分类专家。

#### 常见误区

1. **数据分类未执行**：制定了分类标准但未实际执行标记
2. **DLP 部署不全**：只覆盖邮件，未覆盖云存储、终端等渠道
3. **DSAR 流程不通**：收到用户数据请求后无法在时效内完成响应

#### 验证方法

- 数据分类覆盖率抽查
- DLP 策略有效性测试
- DSAR 响应流程演练

#### 运行指标

- 数据分类完成率（关键数据）
- DLP 策略覆盖率
- DSAR 响应 SLA 达成率
- 数据泄露事件数量

### 9. 安全运营（SOC）

安全运营中心（SOC）负责实时监控、事件响应和威胁狩猎，是安全组织的"值班室"。

如果说其他安全职能是"建设防线"，SOC 就是"守卫防线"。防火墙规则再完善，也需要有人监控是否有异常流量；代码再安全，也需要有人检测是否有入侵行为。SOC 是安全组织中唯一需要 7×24 小时运转的职能，这意味着需要轮班制度、需要应对告警疲劳、需要在压力下快速决策。SOC 分析师常被比喻为"安全医生"——要在海量症状（告警）中快速识别真正的疾病（威胁），既不能漏诊（错过真实攻击），也不能过度诊断（误报导致资源浪费）。

#### 核心职责

**实时监控与告警分类**

实时监控的核心挑战是「告警疲劳」。解决策略是分层过滤，第一层自动过滤已知误报，第二层 L1 分析师快速分类，第三层 L2 深度分析。成熟的 SOC 需预留时间用于告警质量优化。

监控工具的选择需权衡覆盖广度与运营复杂度。实用策略是渐进式覆盖，先部署 SIEM 打好基础，再根据能力增加 EDR、NDR 覆盖。

**安全事件响应与处置**

事件响应遵循 NIST 标准的六个阶段，准备、检测与分析、遏制、根除、恢复、事后复盘。其中检测与分析最耗时，需要收集日志、关联多个数据源、排除干扰信息。遏制措施需权衡速度与业务影响。事后复盘是最容易被忽略但价值极大的阶段。

事件分级需平衡响应速度与资源消耗：

- `P0 (Critical)`，业务已中断或数据已泄露，需立即响应
- `P1 (High)`，攻击进行中但业务未受影响，需快速响应
- `P2 (Medium)`，可疑行为，需在数小时内响应
- `P3 (Low)`，低风险或合规性事件，可在工作日内响应

**主动威胁狩猎**

威胁狩猎与告警监控的区别在于，前者是主动寻找未被检测规则覆盖的威胁。核心方法是「假设驱动」，基于攻击者可能的行为特征进行主动搜索。

威胁狩猎的时间投入需在深度与广度之间权衡，深度狩猎针对特定假设彻底调查，广度狩猎快速扫描多个假设。成熟的 SOC 会将威胁狩猎制度化。

**SOAR 自动化运营**

SOAR 的价值在于将重复性工作自动化。决策标准是 ROI 计算，比较手工处理成本与自动化开发及运行成本。自动化优先级应聚焦高频、人工成本高、自动化难度低的场景。

#### 适用边界

SOC 职能在有实时监控需求的组织中必要。对于安全团队规模小的组织，可外包给 MSSP。

#### 关键约束

- **轮班疲劳约束**：7×24 轮班对分析师身心健康影响大，需考虑补贴和休息周期
- **告警 SLA 约束**：多起事件并发时可能无法达到 SLA，需分级响应
- **日志留存成本约束**：SIEM 存储成本高，需分层存储（热数据在 SIEM，冷数据归档）
- **技能稀缺约束**：优秀的 L3 分析师市场稀缺

#### 团队配置参考

**分层架构**：

- L1 Analyst（初级），告警分类、初步分析、执行 Playbook
- L2 Analyst（中级），深度分析、事件响应、Playbook 开发
- L3 Analyst（高级），复杂事件、威胁狩猎、安全研究

**轮班模式**：

- 12×5（初创期）：工作日值班，夜间靠告警通知
- 7×24（成长期）：3 班倒
- Follow-the-Sun（成熟期）：全球多个 SOC 接力值班

**初创期**：3-5 人，或外包给 MSSP。

**成长期**：SOC Manager、9-15 个 Analyst、SOC Engineer。

**成熟期（全球业务）**：SOC Director、多个地域 SOC Manager、24-45 个 Analyst、威胁狩猎专家、取证分析师。

#### 常见误区

1. **告警积压**：告警处理不及时导致积压，真正的威胁被淹没
2. **复盘缺失**：事件处置完成后不做复盘，同样的攻击可能再次成功
3. **过度依赖工具**：完全依赖 SIEM 告警，缺少主动狩猎能力

#### 验证方法

- 红队演练验证检测能力
- 告警处理时效抽查
- 事后复盘执行率检查

#### 运行指标

- MTTD（平均检测时间）
- MTTR（平均响应时间）
- 告警处理率
- 误报率
- 威胁狩猎发现率

### 10. 红队与攻防

红队职能负责从攻击者视角验证防御有效性，通过模拟真实攻击发现安全体系的薄弱环节。

红队的价值在于"证伪"——安全团队可以声称防御体系固若金汤，但只有红队能证明是否真的如此。红队与蓝队（SOC）的关系类似于"免疫系统训练"：红队模拟病毒入侵，蓝队练习检测和清除，整个安全体系在对抗中不断进化。值得注意的是，红队发现的问题越多，说明安全体系提升的空间越大，这是好事而非坏事。成熟的安全组织不会把红队演练的结果作为考核蓝队的指标，而是作为共同改进的起点。

#### 核心职责

**红队演练**

- 全链路攻击模拟（从外网到内网到数据窃取）
- APT 模拟演练
- 社会工程学测试
- 物理渗透

**渗透测试**

- 应用渗透（Web、API、移动 App）
- 网络渗透
- 无线网络测试
- 专项测试

**紫队协作**

- 与蓝队（SOC）联合演练
- 验证检测规则有效性
- 优化事件响应 Playbook

**BAS（Breach and Attack Simulation）**

- 持续自动化的攻击模拟
- 验证安全控制有效性
- 生成态势评估报告

#### 适用边界

红队职能在以下场景需要内部团队：

- 安全团队规模较大
- 需要频繁进行攻防演练
- 有足够预算支持红队人员培养

对于初创期组织，渗透测试可外包，每年进行 1-2 次。

#### 关键约束

- **红队人才稀缺**：优秀的红队人员市场稀缺且薪资高
- **演练范围控制**：需明确演练边界，避免影响生产系统
- **发现与修复的闭环**：红队发现的问题需要有明确的修复责任人和时间表

#### 团队配置参考

**初创期**：外包渗透测试。

**成长期**：Red Team Lead、2-4 个渗透测试工程师。

**成熟期**：Red Team Director、Senior Penetration Tester、渗透测试工程师团队、社会工程学专家。

#### 常见误区

1. **只测不改**：红队发现问题后无人负责修复
2. **演练范围过窄**：只测应用层，不测社会工程学和物理渗透
3. **与蓝队对立**：红蓝队是对抗而非协作关系，无法形成能力提升闭环

#### 验证方法

- 跟踪红队发现的高危问题修复率
- 评估蓝队对红队攻击的检测率
- 比较历次演练中攻击成功难度的变化

#### 运行指标

- 年度演练频率
- 蓝队检测率
- 高危发现修复率
- 攻击路径阻断时间

---

### 11. 安全工程与自动化

安全工程与自动化是横向支撑职能，通过工具开发和流程自动化提升其他职能的效率。

#### 核心职责

- 安全工具开发（内部工具，如安全评分系统、漏洞管理平台）
- SOAR 编排与自动化
- 安全数据平台（统一安全数据湖）
- CI/CD 安全集成
- 内部安全工具链整合

#### 适用边界

安全工程与自动化职能在安全团队规模较大、有重复性工作需要自动化时必要。对于小型团队，该职能可由其他角色兼任。

#### 团队配置参考

**初创期**：由 AppSec 或 SOC 兼任。

**成长期**：Security Automation Lead、2-4 个安全工程师。

**成熟期**：Director of Security Engineering、平台工程师、自动化工程师、数据工程师。

## 16.3.3 职责边界矩阵（RACI）

职能定义明确后，还需要通过 RACI 矩阵明确典型场景下的协作关系。

RACI 矩阵的价值在于将模糊的"大家一起负责"转化为明确的角色分工。很多组织在没有 RACI 的情况下运作，表面上"和谐协作"，实际上遇到问题时互相推诿或重复工作。一个典型场景：发生安全事件后，SOC、AppSec、InfraSec 三个团队同时开始调查，各自收集证据，但没有人负责汇总信息和协调行动。明确的 RACI 矩阵能避免这种混乱——SOC 是 Responsible（执行者），战略团队是 Accountable（负责人），其他团队是 Consulted（咨询对象）或 Informed（知情人）。

### 场景 1：新应用上线

| 活动     | 战略        | GRC | 架构        | AppSec      | InfraSec    | CloudSec    | 数据安全    | SOC         | 红队        |
| -------- | ----------- | --- | ----------- | ----------- | ----------- | ----------- | ----------- | ----------- | ----------- |
| 威胁建模 | I           | I   | **R** | C           | C           | C           | C           | I           | -           |
| 安全评审 | I           | C   | **A** | C           | C           | C           | C           | I           | -           |
| 代码扫描 | -           | -   | I           | **R** | -           | -           | -           | -           | -           |
| 渗透测试 | -           | I   | C           | C           | -           | -           | -           | C           | **R** |
| 配置基线 | -           | C   | C           | -           | **R** | **R** | -           | -           | -           |
| DLP 策略 | -           | C   | I           | -           | -           | -           | **R** | -           | -           |
| 监控配置 | -           | I   | C           | I           | I           | I           | I           | **R** | -           |
| 上线审批 | **A** | C   | C           | I           | I           | I           | I           | C           | -           |

**RACI 说明**：

- R (Responsible)：执行者
- A (Accountable)：负责人，最终决策者
- C (Consulted)：咨询对象
- I (Informed)：知情人

### 场景 2：安全事件响应

| 活动     | 战略        | GRC         | 架构        | AppSec | SOC           | 红队 | 工程 |
| -------- | ----------- | ----------- | ----------- | ------ | ------------- | ---- | ---- |
| 事件检测 | I           | I           | -           | -      | **R**   | -    | -    |
| 初步分析 | I           | I           | C           | C      | **R**   | -    | -    |
| 深度调查 | I           | C           | C           | C      | **A/R** | C    | C    |
| 遏制措施 | I           | C           | C           | C      | **R**   | -    | C    |
| 根除威胁 | I           | C           | C           | C      | **R**   | -    | C    |
| 高管汇报 | **A** | C           | -           | -      | C             | -    | -    |
| 监管通知 | C           | **R** | -           | -      | C             | -    | -    |
| 事后复盘 | **A** | C           | C           | C      | **R**   | C    | C    |
| 控制改进 | C           | C           | **R** | C      | C             | -    | C    |

### 场景 3：合规认证（ISO 27001）

| 活动     | 战略        | GRC           | 架构        | AppSec      | InfraSec    | 数据安全    | SOC         |
| -------- | ----------- | ------------- | ----------- | ----------- | ----------- | ----------- | ----------- |
| 差距分析 | I           | **R**   | C           | C           | C           | C           | C           |
| 政策制定 | C           | **A/R** | C           | C           | C           | C           | C           |
| 控制实施 | C           | **A**   | **R** | **R** | **R** | **R** | **R** |
| 证据收集 | I           | **R**   | C           | C           | C           | C           | C           |
| 审计应对 | **A** | **R**   | C           | C           | C           | C           | C           |

## 16.3.4 职能演进路径

不同规模的组织，职能配置的优先级不同。

职能配置不是一成不变的，而是随着组织发展逐步演进。初创公司不需要也负担不起完整的安全职能体系，但随着业务增长、监管要求增加、技术架构复杂化，原本可以合并的职能需要独立出来，原本可以外包的职能需要内部化。职能演进的节奏应该与业务发展同步——过早建设会造成资源浪费，过晚建设则会导致安全债务累积。下面的阶段划分是参考建议，实际决策需要结合具体业务场景。

### 阶段 1：初创期（安全团队 1-5 人）

**核心原则**：资源有限，聚焦最关键的职能。

**P0（必须有）**：

- 安全战略与治理（CISO 兼任）
- GRC 与合规（应对客户安全问卷和基础认证）
- 应用安全（代码扫描、漏洞修复）
- 基础设施安全（与 IT 合并）

**P1（可外包或延后）**：

- SOC（外包给 MSSP）
- 架构（CISO 兼任，重大项目请外部咨询）
- 红队（每年外包渗透测试）

**P2（暂不需要）**：

- 产品安全（除非是 B2B SaaS）
- 云安全（与 InfraSec 合并）
- 数据安全（与 GRC 合并）
- 自动化（规模太小）

**典型 5 人团队配置**：

1. CISO（战略 + 架构 + 协调）
2. GRC Specialist（合规 + 政策 + 数据安全）
3. AppSec Engineer（应用安全 + DevSecOps）
4. Infrastructure Security Engineer（网络 + 终端 + 云 + IAM）
5. SOC Analyst（监控 + 事件响应，或外包）

### 阶段 2：成长期（安全团队 10-30 人）

**核心原则**：业务快速增长，安全团队要跟上节奏。

**扩展职能**：

- 独立架构团队，应对复杂系统设计
- 独立云安全团队，云资源快速增长
- 独立数据安全与隐私，跨境业务合规需求
- 7×24 SOC，全球化业务需要全天候监控
- 内部红队，定期验证防御能力

**典型 20 人团队配置**：

- 战略与治理：3 人
- GRC：3 人
- 架构：2 人
- AppSec：5 人
- InfraSec：3 人
- CloudSec：2 人
- 数据安全：2 人
- SOC：8 人（3 班倒）
- 红队：2 人

### 阶段 3：成熟期（安全团队 50-100+ 人）

**核心原则**：多业务线、多地域、复杂合规要求。

**完整职能覆盖**：

- 所有十大职能都有独立团队
- Hub & Spoke 模式（中央团队 + BISO）
- 产品安全团队（如适用）
- 安全工程与自动化
- 全球 SOC (Follow-the-Sun)

## 16.3.5 跨职能协作机制

职能划分清晰只是第一步，有效的协作机制才能让各职能形成合力。安全工作的特点是高度关联性——一个安全事件可能同时涉及网络、应用、数据、合规等多个领域，需要多个职能协同响应。没有协作机制的职能体系就像各自为战的军队，虽然每个单位都很强，但无法形成整体战斗力。

### 日常协作

**Daily Security Standup**：

- 参与：SOC、AppSec、InfraSec、CloudSec（谁当天有事谁参加）
- 频率：每日 15 分钟
- 内容：过去 24 小时安全事件、需要其他团队支持的事项、当天重点监控

**Weekly Security Leadership Sync**：

- 参与：CISO + 各职能 Manager/Director
- 频率：每周 1 小时
- 内容：关键项目进展、资源需求、风险升级

### 周期性协作

**Monthly Security Review**：

- 参与：全体安全团队 + 关键利益相关方
- 频率：每月
- 内容：安全指标回顾、重大事件复盘、下月计划

**Quarterly Business Review**：

- 参与：安全领导层 + 高管
- 频率：每季度
- 内容：OKR 达成情况、预算执行情况、业务风险评估

### 专项协作机制

**安全设计评审（SDR）流程**：

触发条件：任何重大技术变更，包括新系统上线、架构变更、数据流变更、权限变更。

流程：

1. 业务/产品/研发提交 SDR 申请
2. 架构团队指派负责人
3. 架构师做威胁分析
4. 邀请相关职能团队参与评审
5. 出具安全评审报告
6. 跟踪控制措施落地
7. 战略团队审批上线

SLA 参考：

- 低风险项目，3-5 个工作日
- 中等复杂度，1-2 周
- 高风险/复杂项目，2-4 周

**事件响应（IR）协作流程**：

P0/P1 事件流程：

1. SOC 初步分级
2. 拉起 War Room
3. 成立 IR 小组（IR Lead、技术调查组、遏制组、业务联络、合规评估、高管沟通）
4. 定期同步进展
5. 事件关闭
6. 事后复盘

## 16.3.6 职能成熟度指标

### 核心指标体系

| 职能                 | 指标                     | 说明                         |
| -------------------- | ------------------------ | ---------------------------- |
| **战略与治理** | 安全战略与业务战略对齐度 | 业务负责人对安全支持的认可度 |
|                      | 安全项目按时交付率       | 安全项目是否按计划完成       |
| **GRC**        | 合规认证覆盖率           | 关键合规要求是否达成         |
|                      | 审计发现项关闭率及时效   | 审计发现的问题是否及时整改   |
| **架构**       | 重大项目安全评审覆盖率   | 重大项目是否做了 SDR         |
|                      | 威胁建模覆盖率           | 高风险项目是否做了威胁建模   |
| **AppSec**     | 应用安全测试覆盖率       | 应用定期做 SAST/DAST 的比例  |
|                      | 高危漏洞修复 SLA 达成率  | 高危漏洞是否按 SLA 修复      |
| **InfraSec**   | 终端 EDR 覆盖率          | 终端安装 EDR 的比例          |
|                      | 补丁及时率               | 高危补丁按时部署的比例       |
| **CloudSec**   | 云配置合规率             | CIS Benchmark 合规比例       |
| **数据安全**   | 数据分类完成率           | 关键数据已分类标记的比例     |
|                      | DSAR 响应 SLA 达成率     | 用户数据请求按时响应的比例   |
| **SOC**        | MTTD（平均检测时间）     | 从攻击发生到检测的平均时间   |
|                      | MTTR（平均响应时间）     | 从检测到遏制的平均时间       |
| **红队**       | 年度演练频率             | 全链路红队演练次数           |
|                      | 检测率                   | 红队攻击被蓝队检测的比例     |

## 常见挑战与应对

再完善的职能设计也会在实践中遇到挑战。以下是安全组织中最常见的三类问题及应对策略。这些挑战并非设计缺陷，而是组织运作的固有张力——不同职能有不同的视角和优先级，冲突是正常现象，关键在于如何通过机制化解冲突。

### 挑战 1：职能边界模糊

**症状**：AppSec 和 CloudSec 都认为容器安全是对方负责，最后没人管。

**应对方案**：

1. 制定详细的职责划分文档
2. 使用 RACI 矩阵明确典型场景的分工
3. 设立联络人机制，定期同步边界问题
4. 季度复审职责边界

### 挑战 2：职能孤岛

**症状**：SOC 检测到威胁通知 AppSec 修复，但 AppSec 优先级排满，漏洞长期未修复。

**应对方案**：

1. 建立强制性协作流程（正在被利用的漏洞自动升级为 P0）
2. 跨职能 OKR（SOC 和 AppSec 的 OKR 绑定）
3. 定期跨职能演练
4. 领导层推动协作文化

### 挑战 3：资源分配不均

**症状**：SOC 人员充足但告警量少，AppSec 人手不足但需求积压。

**应对方案**：

1. 基于业务风险重新评估资源分配
2. 使用外部资源平衡（MSSP、外包）
3. 内部培训与转岗
4. 自动化释放人力

## 本节小结

安全团队职能划分需要解决"谁负责什么"的问题。十大核心职能覆盖安全工作的全生命周期：战略与治理、GRC、架构、产品安全、应用安全、基础设施安全、云安全、数据安全与隐私、SOC、红队。使用 RACI 矩阵明确职责边界可避免"三不管"和重复工作。职能配置应随组织规模演进：初创期聚焦 4-5 个核心职能，成长期扩展到全职能，成熟期完善 Hub & Spoke 模式。跨职能协作机制（日常同步、周期性回顾、专项流程）是职能体系有效运转的保障。

职能划分不是一次性工作，而是需要持续优化的动态过程。业务变化、技术演进、监管更新都可能导致职能边界需要调整。建议每季度进行一次职能健康度评估，收集内部反馈，识别职能重叠或空白，并根据实际情况动态调整。一个设计精良但僵化不变的职能体系，最终会被业务现实所抛弃。
