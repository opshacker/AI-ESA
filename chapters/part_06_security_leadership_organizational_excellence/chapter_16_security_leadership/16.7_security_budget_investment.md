# 16.7 安全预算与投资

安全预算管理是 CISO 核心职责之一，其挑战在于，安全投资的回报难以直接量化——防止数据泄露与业务收入增长不同，属于"损失规避"而非"价值创造"。本节从预算编制方法、成本结构分析、投资回报论证、多年规划到采购决策，构建完整的安全财务管理框架。

## 16.7.1 安全预算编制

### 预算论证的核心困境

每年预算周期，CISO 面临的根本问题是，如何向 CFO 证明安全投资的必要性？

业务部门的预算逻辑相对清晰——市场投入与收入增长的关联、研发投入与产品交付的对应。安全部门的价值主张则是"防止损失"，投资 SOC 能降低数据泄露风险，但如果不投资，泄露事件也未必发生。这种"反事实"逻辑在财务审批中天然处于劣势。

**适用边界**：本节方法论适用于已建立基本安全职能的组织（安全团队 3 人以上）。初创期企业可简化流程，仅保留合规驱动预算与基线增量方法。

### 预算结构分解

安全预算的典型构成可分为五个主要类别，各类别占比因组织规模与成熟度而异。

人员成本通常占总预算的 45%-55%，是最大单项支出。以 15 人安全团队为例，计入薪酬、福利、招聘与培训费用后，年度人员成本可能达到总预算的一半以上。这一比例在安全成熟度较低的组织中更高，因为工具自动化程度低，依赖人工完成大量工作。

工具与许可费用占 25%-35%。SIEM 平台按日志量收费，EDR 按终端数计费，云安全工具按资源数量收费——这些许可模式导致成本随业务规模线性增长，有时甚至超线性增长（日志量增长通常快于业务增长）。

外部服务与咨询占 8%-12%，包括渗透测试、安全审计、应急响应咨询等。自有团队无法覆盖的专业领域需要外部支持。

合规认证费用占 5%-8%。ISO 27001 认证、SOC 2 审计、PCI DSS 评估等均有明确费用，且需要周期性续费或复审。

项目与基础设施占 5%-10%。架构升级、网络隔离改造、零信任部署等一次性投入归入此类。

| 类别           | 典型占比 | 成本驱动因素                   |
| -------------- | -------- | ------------------------------ |
| 人员成本       | 45-55%   | 团队规模、地区薪资水平、流失率 |
| 工具与许可     | 25-35%   | 日志量、终端数、云资源规模     |
| 服务与咨询     | 8-12%    | 渗透测试频率、审计范围         |
| 合规与认证     | 5-8%     | 认证数量、审计周期             |
| 项目与资本支出 | 5-10%    | 架构改造、一次性部署           |

上述比例为经验参考，实际分配需根据组织的成熟度阶段调整。初创期组织人员占比可能达到 60%-70%，成熟期组织工具占比可能提升至 40%。

### 四种预算编制方法

不同的预算编制方法适用于不同的组织情境，选择依据包括：组织风险管理成熟度、CFO 对安全投资的理解程度、历史预算基础。

#### 方法一：基于风险的预算

基本逻辑：列出业务资产，评估每个资产面临的威胁，量化年度期望损失（ALE = 单次损失 × 年发生概率），将预算投向风险最高的领域。

实际约束：风险量化的精度有限。多数组织的风险评估只能区分"高中低"三档，无法计算精确的 ALE 值。品牌损失、客户流失等间接影响尤其难以量化——数据泄露后的客户流失率是 10% 还是 30%？股价下跌算不算直接损失？这些问题在不同案例中答案差异巨大。

适用场景：已建立成熟风险管理框架的大型组织，尤其是金融机构。如果组织连基础资产清单都尚未建立，不建议采用此方法。

常见误区：

1. 误以为 FAIR 等框架能产出"精确"的风险数值，实际上其输出是概率分布而非确定值
2. 将风险评估等同于预算分配，忽视战略优先级与组织能力约束

#### 方法二：基于合规的预算

基本逻辑：根据 GDPR、HIPAA、PCI DSS 等法规要求，计算满足合规的最小投资，实现"刚好及格"。

实际约束：合规预算容易低估流程建设成本。工具采购只是合规的一部分，流程文档、变更管理、事件响应演练等都需要人力投入。合规审计失败后的返工成本通常是初始预算的 30%-50%。

适用场景：高度监管行业（金融、医疗）的基础预算设定。但即使在这些行业，也不应追求"刚好及格"。

常见误区：

1. 仅计算工具采购费用，忽视实施、培训、流程建设成本
2. 以"通过审计"为唯一目标，忽视实际安全能力建设

验证方法：合规预算的验证标准是能否通过目标认证的差距评估 (gap assessment)。建议在正式审计前进行内部预审或邀请外部顾问做差距分析。

#### 方法三：基线加增量

基本逻辑：以去年预算为基线，根据业务增长比例申请增量。逻辑简单，财务部门易于理解。

实际约束：这是多数组织采用的方法，但容易导致"惯性预算"——历史支出不论有效与否都会延续。实际操作中，部分工具许可证多年未使用但持续续费的情况并不罕见。

适用场景：稳态运营期的年度预算。但建议每 2-3 年进行一次全面审查，清理历史包袱。

常见误区：

1. 假设去年的预算分配是合理的，不审视投资有效性
2. 增量仅跟随业务增长比例，未考虑安全风险的非线性变化

运行指标：基线加增量法需配合工具利用率监控——定期审查各工具的登录频率、功能使用率，识别冗余许可证。

#### 方法四：零基预算

基本逻辑：每年从零开始，论证每一项投资的必要性。消除历史包袱，优化资源分配。

实际约束：工作量巨大，需要 2-3 个月准备详细的投资论证材料。更重要的是，这一过程可能引发团队焦虑——"是不是要裁员了？"——影响士气与稳定性。

适用场景：重大转型时期（新 CISO 上任、组织架构重组、经济衰退期大幅降本）。不建议作为常态化预算方法。

常见误区：

1. 低估零基预算对团队士气的影响
2. 忽视隐性成本（人员重新学习新工具、流程重建的效率损失）

关键约束：

- 零基预算的执行周期通常需要 3-6 个月，在快节奏业务环境中可能错过采购窗口
- 零基审查期间，安全能力可能出现短暂下降（旧工具下线、新工具尚未就绪）

### 预算方法选择矩阵

| 方法       | 适用场景          | 主要约束           | 验证机制         |
| ---------- | ----------------- | ------------------ | ---------------- |
| 基于风险   | 大型金融/医疗组织 | 风险量化精度有限   | 外部风险评估验证 |
| 基于合规   | 强监管行业        | 易低估流程成本     | 合规差距评估     |
| 基线加增量 | 稳态运营期        | 惯性预算风险       | 工具利用率审计   |
| 零基预算   | 重大转型期        | 工作量大、影响士气 | 成本效益分析     |

## 16.7.2 成本结构与行业参考

### 安全预算占比的影响因素

安全预算占 IT 预算的比例受多重因素影响，不同行业、规模、监管环境下的差异显著。

行业因素：金融、医疗等强监管行业的安全投入占比通常高于零售、制造业。这一差异源于监管罚款的威慑效应——金融机构的合规违规罚款可能达到营收的数个百分点，而零售业历史上的处罚力度相对较轻（尽管近年勒索软件攻击正在改变这一格局）。

规模因素：大型企业的安全预算绝对值高，但占比未必更高。中小企业由于缺乏规模效应，人均安全成本反而更高——一个 500 人的企业与一个 5000 人的企业，都需要基础的 SOC 能力、合规认证，但前者无法像后者那样分摊固定成本。

云化程度：云原生企业的安全成本结构与传统企业不同。云服务商承担了基础设施安全责任，企业安全投入集中在应用层与配置管理，工具占比可能高于人员占比（更依赖 CSPM、CWPP 等自动化工具）。

不同行业的典型特征：

- 科技 / SaaS：IT 预算占营收比例高，但安全占 IT 预算的比例相对较低；工具自动化程度高，DevSecOps 成熟度领先
- 金融服务：监管驱动投资，合规成本占比显著；人员配置密集，往往设有专门的风险量化团队
- 医疗健康：整体 IT 预算占比低，安全在有限池子中争取资源；HIPAA 合规要求大量人工核查
- 零售电商：成本敏感型行业，历史安全投入偏低；近年因勒索软件威胁显著增加投入
- 制造业：OT 安全需求上升，正从"合规最小化"向"主动防御"转型

关键约束：行业基准数据仅供参考，不应作为预算目标。同一行业内企业的差异可能大于行业间差异——一家专注 B2B 的科技公司与一家消费者导向的科技公司，风险敞口完全不同。

### 安全团队规模规划

团队规模规划的核心问题是，多少人够用？

行业经验值（安全团队与员工总数比例）为 1:40 到 1:100，但这一范围过于宽泛，实际规划需考虑以下变量：

自动化程度：工具自动化程度高的组织，人员需求相对较低。某互联网企业 8 万员工配置 800 人安全团队（1:100），高于"1:40"的经验值上限，但其自动化覆盖率极高，人员主要从事威胁研究、架构设计等高价值工作，而非日常运维。

外包策略：部分组织将 SOC 运营外包给 MSSP，内部团队专注策略与架构。这种模式下内部团队规模较小，但总安全人力（内部 + 外包）需综合计算。

业务复杂度：业务线数量、地域分布、产品类型多样性都影响安全工作量。同等员工规模下，一家全球分布 15 个办公室、提供 50 种产品的企业，与一家单一办公室、单一产品的企业，安全团队规模需求差异显著。

不同规模企业的典型配置：

| 企业规模               | 典型团队规模 | 组织模式     | 关键考量             |
| ---------------------- | ------------ | ------------ | -------------------- |
| 初创期（50-500 人）    | 3-10 人      | 集中式       | 依赖云服务商安全能力 |
| 成长期（500-2000 人）  | 10-30 人     | 集中式转混合 | 开始嵌入 BISO        |
| 中型（2000-5000 人）   | 30-80 人     | Hub & Spoke  | 核心团队+分布式伙伴  |
| 大型（5000-20000 人）  | 80-300 人    | 分布式+中心  | 全球协调机制         |

验证方法：团队规模是否合理，可通过运营指标检验——人均负责系统数、安全需求响应时效、关键岗位填充率。如果分析师超负荷工作（每人每天处理告警数超出合理阈值）或需求响应周期过长，说明人力不足。

### 工具成本结构

工具采购是安全预算中最容易"失控"的类别。SIEM 厂商按日志量收费，日志量每年增长 30%，费用自然水涨船高；云安全工具按资源数计费，云迁移加速后费用翻倍；各种"最佳实践"工具层层叠加，最终发现工具数量超过 50 个，每个工具的利用率不足 30%。控制工具成本的前提是理解其结构。

工具预算的分配反映安全能力建设的优先级。以下为典型的工具预算分配参考：

SIEM/SOC 平台（占工具预算 20%-25%）：按日志量收费，成本随业务规模增长。日志保留期限延长、数据源增加都会推高成本。

云安全平台（15%-20%）：CSPM/CWPP/CNAPP 类工具按云资源数量收费。多云环境下可能需要多套工具或统一平台。

应用安全工具（15%）：SAST/DAST/SCA 工具链，按代码仓库数量或扫描频率计费。

身份与访问管理（10%-12%）：SSO、MFA、PAM 等按用户数计费。特权访问管理 (PAM) 单价较高。

网络安全（10%）：防火墙、SASE、云代理等。传统网络安全投入占比呈下降趋势。

数据安全与 GRC（10%-15%）：数据分类、隐私合规、GRC 平台等。

常见误区：

1. 仅计算许可费，忽视实施集成成本（API 对接、数据标准化可能耗费许可费 30%-50% 的额外成本）
2. 忽视运维成本（专业服务、培训、内部管理人力）
3. 未预留扩容空间（日志量增长导致年度成本超预算）

运行指标：工具投资有效性可通过利用率监控——功能使用率低于 30% 的工具应纳入整合或淘汰评估。

## 16.7.3 安全投资回报率（ROSI）

### ROSI 计算的本质局限

安全投资回报率（ROSI，return on security investment）的计算公式在理论上清晰：

```
ROSI = (风险缓解价值 - 安全投资成本) / 安全投资成本 × 100%
风险缓解价值 = ALE_Before - ALE_After
ALE（年度期望损失）= SLE（单次损失）× ARO（年发生率）
```

但实际应用中，每个变量都存在显著不确定性：

- 单次损失 (SLE)：数据泄露的平均损失在不同案例中差异巨大，取决于泄露数据类型、数量、监管环境、企业规模、品牌影响等因素
- 年发生率 (ARO)：安全事件概率难以精确估计，行业统计数据只能提供参考区间
- 缓解效果：某工具"能阻止 70% 的攻击"的说法通常来自厂商营销材料，实际效果取决于部署质量与运营能力

关键约束：ROSI 计算的误差范围可能达到 50%-100%。这并非方法论缺陷，而是安全领域的固有特征——与市场营销投入的可追踪转化率不同，安全投入的效果体现为"未发生的损失"，本质上是反事实推理。

### ROSI 的实用价值

尽管存在精度局限，ROSI 计算在以下场景仍有价值：

方案比选：在多个安全项目竞争有限预算时，ROSI 可提供相对排序依据——即使绝对值不精确，不同方案的相对收益差异仍有参考意义。

沟通工具：将技术投资转化为财务语言，有助于与 CFO、董事会的对话。

思维框架：强制进行风险量化思考，避免"拍脑袋"式预算。

### ROSI 计算示例

以下示例展示 ROSI 计算的逻辑框架，数值仅为说明目的（示例口径）。

场景：部署威胁检测平台 (SIEM + SOC 团队)

投资成本估算（3 年）：

- 平台许可费：根据日志量估算
- 实施与集成：通常为许可费的 30%-50%
- 人员成本：分析师团队薪酬
- 培训与基础设施：初始投入较高，后续递减

风险缓解价值估算：

识别主要威胁场景（数据泄露、勒索软件、内部威胁等），分别估算：

- 单次损失期望值（基于历史案例或行业参考）
- 当前发生概率（基于组织风险评估）
- 部署后发生概率（基于检测能力提升假设）

计算示例框架：

```
投资前年度期望损失 = Σ（各威胁场景 SLE × ARO_Before）
投资后年度期望损失 = Σ（各威胁场景 SLE × ARO_After）
风险缓解价值 = （投资前 - 投资后）× 投资年限
ROSI = （风险缓解价值 - 总投资）/ 总投资 × 100%
```

核心假设的敏感性分析：

由于各变量存在不确定性，建议进行敏感性分析——测试关键假设变化 ±30% 时 ROSI 的波动范围。如果即使在悲观假设下 ROSI 仍为正值，项目可行性更强。

常见误区：

1. 采用厂商提供的"最佳情况"数据作为基准假设
2. 未披露关键假设，导致 CFO 质疑数据可信度
3. 仅呈现单一 ROSI 数值，未展示不确定性范围

### 能力指标比 ROSI 更可信

实际项目经验表明，CFO 对 ROSI 计算的精确数值往往持怀疑态度，但对能力提升的具体指标更能接受：

| ROSI 论证               | 能力指标论证                  |
| ----------------------- | ----------------------------- |
| "投资可避免损失 500 万" | "检测时间从 30 天缩短到 3 天" |
| "风险降低 70%"          | "告警误报率从 90% 降至 20%"   |
| "ROSI 45%"              | "人均处理告警量提升 3 倍"     |

能力指标的优势在于可验证——部署 6 个月后可回顾实际 MTTD/MTTR 变化，与承诺对比。ROSI 则是反事实推理，永远无法验证"如果不投资会损失多少"。

验证方法：建议采用"PoC + 里程碑"模式——首期投资做概念验证，以能力指标（而非 ROSI）作为阶段验收标准，验证效果后再追加投资。

## 16.7.4 多年预算规划

### 三年滚动预算模型

安全能力建设不是一年能完成的事。SOC 从无到有需要 18-24 个月的爬坡期，零信任架构的全面部署可能跨越 2-3 年，高级检测能力的成熟需要持续迭代。如果预算规划只看眼前一年，就会陷入"年年重复基础建设、始终无法达到成熟"的困境。多年预算规划帮助组织建立长期视角，确保安全能力的持续演进。

安全预算规划需要多年视角，原因包括：

- 安全能力建设周期长（SOC 成熟通常需要 18-24 个月）
- 工具合同通常为多年期
- 人才培养需要持续投入
- CFO/董事会需要了解未来投资趋势

滚动预算的基本结构：

基线年 (Year 1)：建立预算基准，明确各类别占比。重点是完成基础能力建设——SOC 平台、核心工具链、团队搭建。

能力深化年 (Year 2)：在基线上增量投资。典型方向包括自动化平台 (SOAR)、高级检测能力 (UEBA)、团队扩充。增长率通常高于 Year 1，因为基础能力就绪后，深化投资的边际收益更高。

优化与创新年 (Year 3)：增长率放缓，重点转向工具整合（减少碎片化）、能力优化、新兴领域探索（如 AI 安全）。

年度预算结构变化示意：

| 类别       | Year 1 | Year 2 | Year 3 |
| ---------- | ------ | ------ | ------ |
| 人员成本   | 55-60% | 50-55% | 50-55% |
| 工具与平台 | 25-30% | 30-35% | 30-35% |
| 服务与培训 | 10-12% | 10-12% | 8-10%  |
| 应急预留   | 3-5%   | 2-3%   | 2-3%   |
| 创新基金   | -      | 2-3%   | 3-5%   |

关键约束：

- 多年规划的精度随时间递减——Year 3 的预算可能偏离 ±20%
- 威胁态势、技术演进、业务变化都可能导致计划调整
- 滚动预算需每年更新，而非一成不变的三年计划

### 预算预留策略

预算 100% 分配的常见后果是，突发事件（勒索软件攻击、重大漏洞应急）发生时无资金响应，被迫临时申请，影响响应速度和 CFO 信任度。

预留结构建议：

应急预留（5%-10%）：用于安全事件响应——外部取证、应急咨询、系统恢复、法律与公关支持。如果年度未使用，可滚入次年或归还财务（后者有助于建立信任）。

创新基金（3%-5%）：用于新技术 PoC、研发实验。快速评估新兴工具而无需走完整采购流程。

常见误区：

1. 应急预留过高（超过 15%），导致资源闲置
2. 应急预留用于常规项目超支，而非真正的应急
3. 未建立创新基金，导致新技术评估被拖延到下一预算年度

验证方法：年末审计预留使用情况——应急预留的实际使用率、创新基金的 PoC 转化率。

### 工具生命周期管理

安全工具的采购、使用、淘汰需要有序规划，避免"工具蔓延"（工具数量膨胀但利用率下降）。

典型生命周期考量：

- 商业 SaaS 工具：合同期通常 1-3 年，需提前 3-6 个月启动续约评估
- 开源工具：无许可费但需计算维护人力成本
- 硬件设备：生命周期 5-7 年，需规划替换预算

工具整合收益：

工具数量过多导致管理复杂度上升、集成成本增加、培训成本重复、告警分散难以关联。

整合目标不是"越少越好"，而是在覆盖度与复杂度之间取得平衡。经验表明，15-20 个工具是多数中型企业的合理区间，超过此数量应启动整合评估。

运行指标：

- 工具利用率（功能使用率低于 30% 应纳入淘汰候选）
- 平均每工具的管理人力投入
- 工具间数据流通程度（孤岛工具应优先整合）

## 16.7.5 与 CFO 沟通

### 预算沟通的常见挑战

很多 CISO 抱怨"CFO 不理解安全"，但问题往往在于沟通方式。安全团队习惯的语言（MTTD、误报率、覆盖率）对财务部门来说是外语。CFO 关心的是：这笔钱花出去，能带来什么回报？能规避什么损失？如何验证？如果安全预算提案无法回答这些问题，被质疑或削减是必然结果。

安全预算申请被质疑或削减，往往源于沟通方式而非投资本身的问题。常见挑战包括：

"避免损失无法证明"：CFO 质疑反事实逻辑——如果不投资，损失也未必发生。

应对策略：

- 承认不确定性（"精确值难以计算，但我们可以估算区间"）
- 提供多层证据（行业数据 + 内部检测记录 + 保守估算）
- 提出验证方式（"我们可以邀请外部审计验证方法论"）

"今年全员降本，安全不能例外"：经济下行期的预算削减压力。

应对策略：

- 接受削减，但提供选择题而非被动接受
- 呈现不同削减方案的风险后果
- 给出明确建议（"我建议方案 A，因为……"）

"只认可真金白银节省，不认可避免损失"：CFO 偏好可验证的成本节省。

应对策略：

- 分离 quick wins（短期可量化节省）与战略投资（长期风险管理）
- 先用 quick wins 建立信任（工具整合节省许可费、自动化减少人力）
- 战略投资的收益采用保守估算，作为附加收益而非核心论据

### 沟通的语言转换

将技术指标转化为财务语言：

| 技术表述                  | 财务表述                                |
| ------------------------- | --------------------------------------- |
| 部署 SIEM 平台            | 投资威胁检测，降低数据泄露风险          |
| MTTD 从 8 小时降至 1 小时 | 安全事件响应速度提升 8 倍，减少业务中断 |
| 告警误报率降至 20%        | 分析师效率提升，相当于节省 2 FTE        |
| 零信任架构部署            | 淘汰 VPN，节省许可费与运维成本          |

### 预算提案结构

向董事会或高管层的预算提案，建议采用以下结构：

第一页：执行摘要——投资总额、核心诉求、业务价值、ROI 概要

第二页：为什么需要增长——外部因素（威胁态势）、内部因素（业务增长、云迁移）、合规压力

第三页：不批准的后果——风险场景量化、合规失败影响、业务影响

第四页：阶段里程碑——分季度交付物、KPI 承诺

第五页：行业对标——与同行比较，证明投资合理性

常见误区：

1. 技术术语过多，CFO 无法理解
2. 仅呈现支出，未呈现价值
3. 承诺过度（"100% 防止数据泄露"），后续无法兑现

## 16.7.6 Build vs Buy 决策

### 自建与采购的决策框架

"我们自己开发一个 SIEM，比买商业产品便宜多了"——这是安全团队常见的误判。自建项目的初始成本确实可能低于商业产品的许可费，但隐藏成本往往被严重低估：持续维护、人员依赖、功能迭代、安全更新。真正适合自建的场景极少，多数组织最终发现：自建的总拥有成本 (TCO) 是商业产品的 2-3 倍，而且质量更差。

"Build vs Buy"是安全工具选型的核心决策之一。两种路径各有适用场景。

采购 (Buy) 的适用条件：

- 需求属于标准化场景（SIEM、EDR、CSPM 等）
- 市场有成熟产品
- 组织安全研发能力有限
- 快速部署优先

自建 (Build) 的适用条件：

- 需求具有强差异化（竞争对手没有的独特能力）
- 市场无满足需求的成熟产品
- 组织有足够的安全研发能力（通常 10+ 人专职研发团队）
- 长期战略投入意愿

决策矩阵：

```
            需求差异化程度
            低           高
       ┌─────────┬─────────┐
    高 │  Buy    │ Build   │
战略   │ (高端)  │ (创新)  │
重要性 ├─────────┼─────────┤
    低 │  Buy    │  外包   │
       │ (标准) │         │
       └─────────┴─────────┘
```

### 自建的成本陷阱

自建项目的常见误区：

低估维护成本：开发成本只是冰山一角，维护成本通常是开发成本的 2-3 倍。初始预算只计划了开发，未计入持续迭代、bug 修复、安全更新、文档维护。

人员依赖风险：核心开发人员离职导致知识断层。自建项目的知识集中度通常高于商业产品（后者有完善文档和供应商支持）。

需求变化：自建项目的灵活性反而是劣势——需求变化时需要重新开发，而商业产品可能已经内置了新功能。

质量问题：为赶进度跳过测试和文档，导致产品质量不达标，最终用户拒绝使用。

典型失败案例特征：

- 初始预算严重低估（实际成本是预算的 2-3 倍）
- 项目周期延长（从 12 个月拖到 24-36 个月）
- 关键人员流失导致项目停滞
- 最终弃用，重新采购商业产品

### 混合策略

多数成熟组织采用混合策略：

- 核心 buy：标准化安全能力（SIEM、EDR、身份管理）采用成熟商业产品
- 差异化 build：真正独特的需求自建（通常不超过工具栈的 5%-10%）
- 开源 + 定制：成本敏感场景采用开源工具 + 内部定制（需要较强运维能力）

关键成功因素：

1. 严格界定 build 的边界——只 build 真正独特的能力
2. Build 项目必须持续投入，不是一次性开发
3. 充分的文档化与知识传承，降低人员依赖风险

## 本节要点

预算编制：四种方法（基于风险、基于合规、基线加增量、零基预算）各有适用场景，多数组织采用基线加增量作为常态方法，定期进行合规或零基审查。

成本结构：人员成本通常占 45%-55%，工具占 25%-35%；具体比例因组织成熟度、自动化程度、外包策略而异。

ROSI 计算：方法论清晰但精度有限，误差范围可达 50%-100%；能力指标（MTTD、误报率等）比 ROSI 数值更可验证、更能获得 CFO 信任。

多年规划：采用滚动预算模型，预留应急与创新基金，规划工具生命周期管理。

CFO 沟通：将技术语言转化为财务语言，用 quick wins 建立信任，战略投资采用保守估算。

Build vs Buy：90% 以上场景应采购成熟产品，仅对真正独特的差异化需求考虑自建。
