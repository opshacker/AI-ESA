# 15.7 AI 安全运营

---

## 引言

本节聚焦 **AI 系统特有的安全运营能力**，是对通用 SOC 运营体系（[第11章 安全运营](../../part_04_security_operations_defense_capabilities/chapter_11_security_operations/README.md)）的 AI 专项扩展。

### 与第11章的定位区分

| 维度 | 第11章 SOC 运营 | 15.7 AI 安全运营 |
|------|----------------|-----------------|
| **定位** | 通用安全运营中心建设与运营 | AI 系统特有的安全服务与响应 |
| **覆盖范围** | 全企业安全事件（网络、端点、身份等） | 仅 AI/ML 系统相关事件 |
| **检测对象** | 传统威胁（恶意软件、入侵、数据泄露） | AI 特有威胁（对抗样本、模型投毒、提示词注入） |
| **响应流程** | 标准 IR 流程（NIST SP 800-61） | AI 专项 Playbook（补充标准 IR 流程） |
| **工具栈** | SIEM、SOAR、EDR、TIP | Guardrails、ModelScan、Garak、ART |
| **组织** | SOC 团队（L1/L2/L3 分析师） | ModelSec 团队 + SOC 协作 |

**协作模式**：AI 安全事件由 SOC 统一受理，涉及 AI 特有威胁的事件升级至 ModelSec 团队处理。15.7 的 Playbook 作为第11章通用 Playbook 库的 AI 专项补充。

### 服务度量指标

| 指标类别 | 具体指标 | 计算方式 | 目标值 | 采集频率 |
|----------|----------|----------|--------|----------|
| **服务可用性** | Guardrails 服务可用性 | 正常运行时间 / 总时间 | ≥ 99.9% | 实时 |
| **响应效率** | 服务请求首次响应时间 | 从提交到首次响应的时间 | P1 < 15min, P2 < 2h, P3 < 8h | 每请求 |
| **交付效率** | 服务请求关闭周期 | 从提交到关闭的时间 | 符合 SLA ≥ 95% | 每请求 |
| **服务质量** | 服务满意度评分 | 用户评分平均值 | ≥ 4.0/5.0 | 每请求 |
| **服务覆盖** | AI 系统安全评估覆盖率 | 已评估系统 / 总 AI 系统 | ≥ 80% | 季度 |
| **培训覆盖** | AI 安全培训覆盖率 | 已培训人员 / 目标人群 | ≥ 90% | 年度 |

### SLA 分层指标体系

| SLA 层级 | 响应时间 | 解决时间 | 升级时间 | 适用事件类型 | 惩罚条款 |
|----------|----------|----------|----------|--------------|----------|
| **P1 Critical** | ≤ 15 min | ≤ 4h | 30min 无进展 | 数据泄露、系统接管、监管事件 | 超时每 15min 通报 CTO |
| **P2 High** | ≤ 1h | ≤ 24h | 2h 无进展 | 提示词注入成功、模型行为异常 | 超时升级 P1 处理 |
| **P3 Medium** | ≤ 4h | ≤ 72h | 8h 无进展 | 单次攻击尝试、轻微违规 | 月度 SLA 达成率考核 |
| **P4 Low** | ≤ 8h | ≤ 5 工作日 | 无 | 安全咨询、非紧急审计 | 季度服务满意度考核 |

**SLA 时钟暂停条件**：
- 等待业务方提供必要信息（需明确记录等待开始/结束时间）
- 变更窗口外的修复操作（需预先获得审批）
- 外部依赖方响应（需有书面请求记录）

**SLA 达成验证**：
```
月度 SLA 达成率 = (SLA 内关闭的工单数 / 总工单数) × 100%
目标: P1 ≥ 95%, P2 ≥ 90%, P3/P4 ≥ 85%
```

- 模型投毒 / 后门检测
- 对抗样本攻击
- 模型窃取 / 逆向工程
- AI 生成内容违规
- 模型性能异常漂移

### Playbook 1: 提示词注入攻击响应

**触发条件**:
- Guardrails 系统检测到高置信度注入攻击
- 用户报告 AI 系统输出异常内容
- 日志分析发现可疑提示词模式

**严重程度评估矩阵**:

| 影响因素 | P1 (Critical) | P2 (High) | P3 (Medium) |
|----------|---------------|-----------|-------------|
| 数据泄露 | 确认泄露客户 PII | 泄露内部敏感信息 | 泄露非敏感信息 |
| 系统控制 | 攻击者获得系统控制 | 绕过访问控制 | 仅绕过内容过滤 |
| 影响范围 | 面向公众的核心系统 | 内部关键系统 | 测试/边缘系统 |

**响应流程**:

```
Phase 1: 遏制 (Target: 15 分钟内)
├── 1.1 启用紧急输入过滤规则
│   └── 命令: guardrails-cli add-rule --pattern "<攻击模式>" --action block
├── 1.2 若确认数据泄露，触发 DLP 应急响应
├── 1.3 考虑是否需要临时下线受影响系统
└── 1.4 通知业务负责人

Phase 2: 调查 (Target: 4 小时内)
├── 2.1 提取攻击日志
│   └── 关键字段: timestamp, user_id, prompt, response, detection_score
├── 2.2 分析攻击向量
│   ├── 直接注入 vs 间接注入（通过 RAG 文档）
│   ├── 攻击变体识别（编码、多语言、角色扮演等）
│   └── 判断是否为针对性攻击 vs 自动化扫描
├── 2.3 影响范围评估
│   ├── 受影响用户数量
│   ├── 泄露数据类型与数量
│   └── 是否存在横向扩展
└── 2.4 根因分析
    └── 检测规则缺失 / System Prompt 设计缺陷 / 权限配置问题

Phase 3: 修复 (Target: 24 小时内)
├── 3.1 更新检测规则覆盖攻击变体
├── 3.2 加固 System Prompt（如适用）
├── 3.3 强化输出过滤规则
├── 3.4 修复后红队验证
└── 3.5 灰度恢复服务

Phase 4: 复盘 (Target: 72 小时内)
├── 4.1 事件报告撰写
├── 4.2 检测规则库更新
├── 4.3 流程改进项识别
└── 4.4 知识库更新
```

**关键工具清单**:

| 工具 | 用途 | 命令/位置 |
|------|------|-----------|
| Guardrails CLI | 紧急规则更新 | `guardrails-cli add-rule` |
| ELK/Splunk | 日志分析 | 索引: `llm-access-logs-*` |
| Garak | 攻击向量验证 | `garak --model <target> --probes injection` |

### Playbook 2: 模型投毒/后门检测响应

**触发条件**:
- 模型扫描工具检测到可疑代码
- 模型行为出现无法解释的异常
- 供应链情报显示上游模型存在风险

**响应流程**:

```
Phase 1: 隔离 (Target: 30 分钟内)
├── 1.1 停止受影响模型的所有推理请求
├── 1.2 隔离模型文件（不要删除，保留取证）
├── 1.3 切换到备用模型或降级服务
└── 1.4 通知 AI 平台团队和业务负责人

Phase 2: 分析 (Target: 8 小时内)
├── 2.1 模型文件静态分析
│   └── 工具: ModelScan, Fickling
├── 2.2 对比模型哈希与官方来源
├── 2.3 行为分析（触发后门的输入模式）
├── 2.4 追溯模型来源
│   └── 下载时间、来源、下载者、审批记录
└── 2.5 评估受影响推理结果范围

Phase 3: 处置
├── 3.1 若确认投毒: 从受信来源重新获取模型
├── 3.2 若后门复杂: 考虑从头训练或更换模型
├── 3.3 更新模型仓库准入扫描规则
├── 3.4 复查其他同来源模型
└── 3.5 更新模型供应链安全政策

Phase 4: 强化
├── 4.1 强制所有模型入库扫描
├── 4.2 建立模型 SBOM 与签名机制
└── 4.3 定期模型完整性巡检
```

### Playbook 3: 模型窃取/逆向工程响应

**触发条件**:
- API 异常高频调用（超出正常业务量级）
- 调用模式符合模型窃取特征（系统性输入探测）
- 竞争对手出现功能相似的产品（需业务情报支持）

**检测信号**:

| 信号类型 | 检测指标 | 阈值建议 |
|----------|----------|----------|
| 流量异常 | 单用户/IP 调用量 | > 正常 10 倍 |
| 模式异常 | 输入分布覆盖度 | 输入空间覆盖度异常高 |
| 行为异常 | 仅请求 logits/embedding | 连续请求无业务上下文 |

**响应流程**:

```
Phase 1: 验证与遏制 (Target: 1 小时内)
├── 1.1 确认是否为误报（合法批量调用 vs 攻击）
├── 1.2 对可疑账户/IP 实施速率限制
├── 1.3 启用输出扰动（如适用）
└── 1.4 通知法务团队（知识产权保护视角）

Phase 2: 深度分析 (Target: 24 小时内)
├── 2.1 分析攻击者调用模式
│   ├── 输入覆盖策略
│   ├── 使用的账户/IP 分布
│   └── 时间模式
├── 2.2 评估信息泄露程度
│   ├── 输出多少推理结果
│   ├── 是否暴露了内部数据
│   └── 模型可替代性评估
└── 2.3 攻击者归因（可选）

Phase 3: 长期缓解
├── 3.1 优化 API 访问控制
│   └── 更严格的认证、授权、审计
├── 3.2 实施模型输出水印
├── 3.3 强化监控规则
└── 3.4 评估是否需要法律手段
```

### Playbook 4: AI 生成内容违规响应

**触发条件**:
- 内容审核系统标记违规输出
- 用户投诉/举报 AI 生成不当内容
- 外部曝光（社交媒体、媒体报道）

**严重程度评估**:

| P1 (Critical) | P2 (High) | P3 (Medium) |
|---------------|-----------|-------------|
| 生成违法内容（暴恐、CSAM 等） | 生成严重偏见/歧视内容 | 生成轻微不当内容 |
| 外部大规模曝光 | 单一用户投诉但内容严重 | 内部发现，未外泄 |
| 监管机构介入 | 可能引发公关危机 | 无外部影响 |

**响应流程**:

```
Phase 1: 紧急遏制 (Target: 15 分钟内 for P1)
├── 1.1 添加紧急内容过滤规则
├── 1.2 若 P1 违规，考虑临时下线
├── 1.3 保全证据（截图、日志、对话记录）
└── 1.4 通知公关/法务团队

Phase 2: 调查
├── 2.1 还原触发违规的对话上下文
├── 2.2 分析根因
│   ├── 用户恶意诱导
│   ├── 训练数据污染
│   ├── 过滤规则缺失
│   └── System Prompt 缺陷
├── 2.3 评估影响范围（其他用户是否可复现）
└── 2.4 若涉及监管，准备报告材料

Phase 3: 修复与沟通
├── 3.1 更新内容过滤规则
├── 3.2 必要时调整 System Prompt
├── 3.3 用户沟通（若适用）
├── 3.4 公关声明（若外部曝光）
└── 3.5 监管报告（若法规要求）

Phase 4: 长期改进
├── 4.1 强化内容安全测试用例
├── 4.2 更新红队测试场景
└── 4.3 培训相关团队
```

## 15.7.4 AI 系统业务连续性与灾难恢复（BCP/DR）

AI 系统的业务连续性与灾难恢复具有其特殊性：模型资产难以快速重建、推理服务对延迟敏感、训练数据恢复成本高昂。本节针对 AI 系统的特有恢复挑战提供专项指导。

### AI 系统 BCP/DR 特殊挑战

| 挑战维度 | 传统 IT 系统 | AI 系统 | 影响分析 |
|----------|-------------|---------|----------|
| **资产恢复** | 应用代码 + 配置 + 数据库 | 模型权重 + 训练数据 + 推理配置 + 特征工程代码 | 模型权重体积大，恢复时间长 |
| **状态重建** | 数据库事务日志回放 | 模型无法"回放"，需重新训练或恢复快照 | 重训练周期可达数天至数周 |
| **一致性** | 数据强一致性 | 模型版本 + 特征 + 推理配置需版本对齐 | 版本不一致导致推理结果异常 |
| **降级方案** | 功能开关、限流 | 回退旧版本模型、规则引擎替代 | 需预置降级模型和规则 |
| **供应商依赖** | 云服务 SLA | 模型 API 提供商（OpenAI/Claude 等）不可用 | 需多供应商策略 |

#### 策略二：多供应商冗余

针对依赖外部 LLM API 的场景：

```python
# Multi-LLM Provider 容灾示例
class ResilientLLMClient:
    """支持多供应商容灾的 LLM 客户端"""

    def __init__(self, providers_config: dict):
        self.providers = self._init_providers(providers_config)
        self.circuit_breakers = {}
        self.fallback_order = providers_config.get("fallback_order", [])

    def _init_providers(self, config):
        return {
            "openai": OpenAIClient(config.get("openai")),
            "anthropic": AnthropicClient(config.get("anthropic")),
            "azure_openai": AzureOpenAIClient(config.get("azure_openai")),
            "local_llm": LocalLLMClient(config.get("local_llm")),  # 本地部署兜底
        }

    async def generate(
        self,
        prompt: str,
        model_preference: str = "openai",
        max_retries: int = 3
    ) -> LLMResponse:
        """带容灾的生成请求"""

        tried_providers = set()

        # 优先使用指定供应商
        providers_to_try = [model_preference] + [
            p for p in self.fallback_order if p != model_preference
        ]

        for provider_name in providers_to_try:
            if provider_name in tried_providers:
                continue
            tried_providers.add(provider_name)

            provider = self.providers.get(provider_name)
            if not provider:
                continue

            # 检查熔断器状态
            if self._is_circuit_open(provider_name):
                self._log_circuit_open(provider_name)
                continue

            try:
                response = await provider.generate(prompt, max_retries=max_retries)
                self._record_success(provider_name)
                return response

            except ProviderUnavailableError as e:
                self._record_failure(provider_name, e)
                self._log_failover(provider_name, str(e))
                continue

            except RateLimitError as e:
                self._record_failure(provider_name, e)
                # 速率限制不触发熔断，但尝试下一个供应商
                continue

        # 所有供应商都失败
        raise AllProvidersUnavailableError(
            "All LLM providers failed",
            tried_providers=list(tried_providers)
        )

    def _is_circuit_open(self, provider: str) -> bool:
        """熔断器检查"""
        cb = self.circuit_breakers.get(provider)
        if not cb:
            return False
        return cb.is_open and not cb.should_attempt_reset()

    def _record_failure(self, provider: str, error: Exception):
        """记录失败并更新熔断器"""
        if provider not in self.circuit_breakers:
            self.circuit_breakers[provider] = CircuitBreaker(
                failure_threshold=5,
                reset_timeout_seconds=300
            )
        self.circuit_breakers[provider].record_failure()
```

## 15.7.5 AI 安全监控指标体系

有效的 AI 安全运营需要数据驱动的度量体系。本节提供涵盖威胁检测、模型安全、合规治理和运营效率四个维度的指标框架，支持 CISO 向董事会汇报和安全团队日常运营。

### 指标体系设计原则

| 原则 | 说明 | 反面案例 |
|-----|------|---------|
| **可采集** | 指标必须可从现有系统自动采集 | 设计"模型安全意识成熟度"但无采集手段 |
| **可行动** | 指标异常时有明确的响应动作 | 仅展示指标但无告警阈值和处置流程 |
| **可对比** | 支持时间维度对比和业界基准对比 | 绝对值无意义，需趋势和基线 |
| **分层** | 高管看趋势，团队看明细 | 向 CEO 汇报 P95 延迟数据 |

### 实时运营告警指标（KRI）

用于 SOC 实时监控，触发即时响应的关键风险指标。

```yaml
# AI 安全告警规则示例 (Prometheus/AlertManager 格式)
groups:
  - name: ai_security_alerts
    rules:
      # 提示词注入突增告警
      - alert: PromptInjectionSurge
        expr: increase(guardrails_injection_blocked_total[5m]) > 50
        for: 2m
        labels:
          severity: warning
          team: modelsec
        annotations:
          summary: "提示词注入攻击激增"
          description: "5分钟内检测到 {{ $value }} 次注入攻击，超过阈值"
          runbook: "https://wiki/playbook/prompt-injection"

      # 模型完整性告警
      - alert: ModelIntegrityFailed
        expr: model_integrity_check_passed == 0
        for: 0s
        labels:
          severity: critical
          team: mlops
        annotations:
          summary: "模型完整性校验失败"
          description: "模型 {{ $labels.model_name }} 完整性校验未通过"
          runbook: "https://wiki/playbook/model-tampering"

      # PII 泄露告警
      - alert: PIILeakageDetected
        expr: increase(llm_output_pii_detected_total[1m]) > 0
        for: 0s
        labels:
          severity: critical
          team: modelsec
        annotations:
          summary: "检测到 PII 泄露"
          description: "LLM 输出中检测到 PII 信息"
          runbook: "https://wiki/playbook/pii-leakage"

      # 模型推理异常
      - alert: ModelInferenceAnomaly
        expr: |
          abs(
            rate(model_inference_latency_seconds_sum[5m]) /
            rate(model_inference_latency_seconds_count[5m])
            - model_inference_latency_baseline
          ) / model_inference_latency_baseline > 0.5
        for: 5m
        labels:
          severity: warning
          team: mlops
        annotations:
          summary: "模型推理延迟异常"
          description: "模型 {{ $labels.model_name }} 延迟偏离基线超过 50%"

      # API 滥用检测
      - alert: APIAbuseDetected
        expr: |
          sum(rate(llm_api_requests_total[5m])) by (user_id)
          > on(user_id) group_left max(user_api_rate_limit) * 0.9
        for: 2m
        labels:
          severity: warning
          team: platform
        annotations:
          summary: "API 使用接近限额"
          description: "用户 {{ $labels.user_id }} API 使用率达到限额的 90%"

      # 模型漂移告警
      - alert: ModelDriftDetected
        expr: model_psi_score > 0.2
        for: 1h
        labels:
          severity: warning
          team: mlops
        annotations:
          summary: "检测到模型漂移"
          description: "模型 {{ $labels.model_name }} PSI 分数超过阈值"
```

### 指标治理与质量

#### 指标元数据管理

| 字段 | 说明 | 示例 |
|-----|------|-----|
| metric_id | 唯一标识 | AI_SEC_001 |
| metric_name | 指标名称 | 提示词注入检测率 |
| definition | 业务定义 | 成功检测的注入攻击占总攻击的比例 |
| formula | 计算公式 | detected_attacks / (detected + bypassed) |
| data_source | 数据来源 | guardrails_detection_log |
| owner | 负责人 | ModelSec Team Lead |
| refresh_frequency | 刷新频率 | 实时 |
| alert_threshold | 告警阈值 | <90% 触发 Warning |
| target_value | 目标值 | ≥95% |
| effective_date | 生效日期 | 2025-01-01 |
| review_cycle | 审查周期 | 季度 |

#### 指标质量检查

| 质量维度 | 检查方法 | 频率 | 责任人 |
|---------|---------|------|--------|
| 完整性 | 检查数据缺失率 | 每日自动 | 数据团队 |
| 准确性 | 抽样人工复核 | 月度 | 安全团队 |
| 及时性 | 监控数据延迟 | 实时 | 平台团队 |
| 一致性 | 跨系统数据比对 | 季度 | 审计团队 |
| 有效性 | 业务价值评估 | 半年 | 安全管理层 |

**指标淘汰标准**：
- 连续 3 个月无人查看
- 无法采集或采集成本过高
- 业务场景已变化，指标失去意义
- 被更有效的指标替代

## 本节要点

AI 安全服务化和 Playbook 化是将安全能力从"被动响应"转向"主动服务"的关键：

- **服务目录**：通过标准化的服务定义和 SLA，让业务团队清晰了解可获得的安全支持
- **事件响应 Playbook**：确保团队在事件发生时快速、一致地响应，减少因人员经验差异导致的处置质量波动
- **持续演练**：通过定期演练验证流程有效性，发现改进机会
