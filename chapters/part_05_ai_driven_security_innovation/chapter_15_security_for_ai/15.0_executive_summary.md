# 执行摘要

> 本节从战略与工程双重视角，阐述 Security for AI 的核心挑战、独特威胁特征与应对框架，帮助安全决策者在 AI 安全投资与风险管理中识别关键权衡点。

## AI 安全战略框架

### 战略目标声明

企业 AI 安全战略应回答三个核心问题：**为什么做**（业务驱动）、**做到什么程度**（成熟度目标）、**如何衡量成功**（度量体系）。

```
AI 安全战略金字塔
┌─────────────────────────────────────┐
│           愿景 (Vision)              │
│   "让 AI 成为可信的业务资产"          │
├─────────────────────────────────────┤
│          使命 (Mission)              │
│  保护 AI 系统免受威胁，确保合规运营    │
├─────────────────────────────────────┤
│        战略目标 (Goals)              │
│  • 风险可控：P0 威胁上线前缓解        │

├─────────────────────────────────────┤
│       战略原则 (Principles)          │

│       能力建设 (Capabilities)        │
│  治理 | 架构 | 工程 | 运营 | 合规    │
└─────────────────────────────────────┘
```

### 战略原则

AI 安全战略应遵循以下核心原则：

| 原则 | 含义 | 落地要求 | 反模式 |
|------|------|----------|--------|
| **风险驱动** | 基于业务影响和威胁可利用性分配资源 | P0/P1/P2 威胁分级，ROI 评估 | 合规驱动（仅满足审计要求） |
| **安全左移** | 将安全嵌入 AI 开发生命周期早期 | ML SDL 集成、设计阶段威胁建模 | 事后补救（上线后再加固） |
| **业务协同** | 安全是业务的促进者而非阻碍者 | 分阶段实施、明确残留风险边界 | 过度防护（安全成为瓶颈） |
| **纵深防御** | 多层防护，单点失效不致全局沦陷 | 输入/模型/输出/运行时多层控制 | 单点依赖（仅靠一道防线） |
| **持续演进** | AI 安全是持续过程而非一次性项目 | 年度预算、季度评审、持续红队 | 一次性投入思维 |

### 业务对齐矩阵

AI 安全能力应与业务目标显式对齐。下表将业务目标映射到所需的安全能力：

| 业务目标 | 安全需求 | 关键能力 | 成熟度要求 | 对应章节 |
|----------|----------|----------|------------|----------|
| **快速上市** | 不因安全阻塞上线 | MVP 防护、分阶段实施 | L2 | 15.4 |
| **进入企业市场** | 满足企业客户安全评估 | 供应链安全、审计能力 | L3 | 15.4, 15.6 |
| **欧盟市场拓展** | EU AI Act 合规 | 风险评估、ISO 42001 | L3-L4 | 15.6 |
| **金融行业部署** | 强合规、高可解释性 | 偏见检测、决策审计 | L4 | 15.5, 15.6 |
| **保护核心 IP** | 防止模型窃取 | 水印、访问控制、推理保护 | L3 | 15.3, 15.4 |
| **用户信任** | 防止数据泄露、避免幻觉 | 输出过滤、隐私保护 | L2-L3 | 15.2, 15.5 |

## AI 安全的本质困境

AI 安全与传统安全存在本质差异：

| 维度 | 传统安全 | AI 安全 |
|-----|---------|--------|
| 基本假设 | 漏洞可修复（工程问题） | 部分风险数学上难以根除 |
| 典型威胁 | 缓冲区溢出、SQL 注入、XSS | 对抗样本、数据投毒、模型记忆 |
| 应对方式 | 补丁、输入验证、参数化查询 | 提高攻击成本、残余风险管理 |
| 承诺边界 | 可承诺"修复漏洞" | 只能承诺"将攻击成本提升到可接受水平" |

以对抗样本攻击为例：这不是工程缺陷，而是源于机器学习模型的数学特性——模型在高维空间中的决策边界天然存在脆弱点，攻击者可以通过精心构造的输入利用这些脆弱点。

实践后果：
- 传统安全工具（防火墙、WAF、DLP）对 AI 特有威胁效果有限
- 防火墙无法拦截 prompt injection，WAF 无法识别对抗样本
- AI 安全需要引入"残余风险"概念，向管理层汇报时必须坦诚技术边界

## AI 安全威胁的三个核心挑战

### 挑战一：对抗样本——数学特性而非工程缺陷

对抗样本攻击的本质：深度学习模型对输入空间的微小扰动高度敏感。学术研究已证明这一特性在数学上难以根除。

适用边界：

| 模型类型 | 受影响程度 | 典型场景 |
|---------|-----------|---------|
| 深度学习分类/检测/识别模型 | 高 | 自动驾驶视觉识别、金融风控评分、安全威胁检测 |
| 传统规则引擎 | 低 | 规则型欺诈检测 |
| 决策树类模型 | 较低 | 简单分类任务 |

关键约束：

| 约束类型 | 具体表现 | 应对策略 |
|---------|---------|---------|
| 准确率-鲁棒性权衡 | 对抗训练可降低攻击成功率，但导致正常样本准确率下降 | 评估业务可接受的准确率下降范围 |
| 计算成本 | 对抗训练需生成对抗样本，训练时间增加；集成防御推理成本成倍增长 | TCO 评估，平衡安全投入与业务收益 |
| 防御时效性 | 新型攻击方法不断涌现，防御措施存在滞后性 | 建立持续更新机制，定期红队测试 |

常见误区：

| 误区 | 事实 | 正确做法 |
|-----|-----|---------|
| 对抗训练是银弹 | 只能防御已知攻击类型，对新型攻击效果有限 | 将对抗训练作为多层防御之一 |
| 忽视业务可行性评估 | 准确率下降可能导致业务拒绝方案 | 实施前评估准确率下降是否在业务可接受范围内 |

验证方法：

| 验证项 | 工具/方法 | 频率 | 验收标准 |
|-------|---------|-----|---------|
| 鲁棒性测试 | IBM ART、CleverHans | 模型更新时 | 攻击成功率低于基线 |
| 准确率监控 | A/B 测试 | 持续 | 准确率在业务可接受范围内 |
| 红队测试 | 定制攻击脚本 | 每季度 | 覆盖主流攻击类型 |

运营指标：
- 对抗攻击成功率：定期红队测试，阈值根据业务风险容忍度设定
- 正常样本准确率：持续监控，设定下限告警（如 <95% 触发审查）

实践权衡：当业务对准确率要求严格时，可考虑组合策略——接受一定对抗攻击风险，但增强异常检测和人工复核环节，而非单纯依赖对抗训练。

### 挑战二：数据投毒——供应链攻击的延伸

训练数据污染是软件供应链攻击在 AI 领域的延伸。攻击者通过污染训练数据，可在模型中植入后门——模型在正常数据上表现正常，但遇到特定触发条件时产生攻击者预期的输出。

适用边界：

| 数据来源 | 风险等级 | 典型场景 |
|---------|---------|---------|
| 互联网爬虫 | 高 | 通用 LLM 预训练 |
| 开源数据集 | 高 | 模型微调 |
| 用户反馈数据 | 中-高 | RLHF、fine-tuning |
| 第三方数据提供商 | 中 | 专业领域模型训练 |
| 内部闭源数据 | 低 | 企业内部模型（仍需防范内部威胁） |

关键约束：

| 约束类型 | 具体表现 | 应对策略 |
|---------|---------|---------|
| 检测成本 | 人工审查大规模训练数据成本高昂；自动检测存在误报和漏报 | 分层审查策略 |
| 溯源困难 | 数据来源复杂时难以追溯每条数据的完整供应链 | 建立数据谱系记录 |
| 延迟影响 | 投毒影响可能在模型上线后很长时间才显现 | 持续监控模型行为异常 |

常见误区：

| 误区 | 事实 | 正确做法 |
|-----|-----|---------|
| 完全信任开源数据集 | 开源平台缺乏统一安全审查机制，存在被恶意污染的风险 | 对开源数据集进行抽样审查和异常检测 |
| 忽视用户反馈数据风险 | fine-tuning 阶段的用户反馈可能被恶意用户利用 | 对用户反馈数据进行异常检测和过滤 |

验证方法：

| 验证项 | 方法 | 频率 | 验收标准 |
|-------|-----|-----|---------|
| 数据抽样审查 | 人工复核随机样本 | 每批次数据 | 异常率 <0.1% |
| 数据溯源 | 记录每条数据的来源和处理链 | 持续 | 100% 可追溯 |
| 异常检测 | 使用离群点检测算法 | 数据入库时 | 自动标记离群样本 |
| 后门检测 | 模型上线前测试 | 模型发布前 | 无已知后门触发 |

运营指标：
- 训练数据审查覆盖率：按模型风险等级设定（高风险 100%，中风险 20%，低风险 5%）
- 数据异常检测告警数：监控训练数据管道中的异常

分层审查策略：

| 模型风险等级 | 审查要求 | 典型场景 |
|-------------|---------|---------|
| 高风险 | 全量人工审查 + 自动检测 | 金融决策、安全控制 |
| 中风险 | 20% 抽样审查 + 自动检测 | 客服机器人、内容推荐 |
| 低风险 | 仅自动检测 | 内部工具、实验项目 |

### 挑战三：隐私泄露——模型对训练数据的记忆

AI 模型会"记忆"训练数据中的信息，且这种记忆难以完全消除。通过特定查询方式可从模型输出中提取训练数据中的敏感信息（如邮箱地址、电话号码、代码片段）。

适用边界：

| 模型类型 | 记忆泄露风险 | 主要攻击方式 |
|---------|-------------|-------------|
| 大语言模型（LLM） | 高 | 提示词攻击诱导输出训练数据 |
| 涉及个人信息的分类模型 | 中 | 成员推断攻击（推断特定个体是否在训练集中） |
| 小型专用模型 | 低 | 模型反演攻击 |

关键约束：

| 约束类型 | 具体表现 | 量化参考 |
|---------|---------|---------|
| 隐私-性能权衡 | 差分隐私的隐私预算（ε）与模型性能呈反向关系 | ε<1 时准确率可能下降 5-15% |
| 合规压力 | GDPR、PIPL 等法规对个人数据处理有严格要求 | 违规罚款可达年收入 4%（GDPR） |
| 技术复杂性 | 差分隐私实施需要专业知识 | 需要 ML 安全专业人员 |

常见误区：

| 误区 | 事实 | 正确做法 |
|-----|-----|---------|
| 差分隐私可解决所有隐私问题 | 差分隐私在实践中面临隐私与性能的艰难权衡 | 评估业务可接受的性能下降范围 |
| 忽视输出端的隐私泄露 | 即使训练时未使用敏感数据，模型输出也可能泄露敏感信息 | 对模型输出进行 DLP 过滤 |

验证方法：

| 验证项 | 工具/方法 | 频率 | 验收标准 |
|-------|---------|-----|---------|
| 成员推断攻击测试 | ML Privacy Meter | 模型发布前 | 攻击成功率 <基线 |
| 提示词攻击测试 | 定制攻击脚本 | 每季度 | 无法诱导输出训练数据 |
| 输出 DLP 扫描 | 正则匹配 + NER | 持续 | 敏感信息检出率 <0.1% |

运营指标：
- 成员推断攻击成功率：定期测试并监控趋势
- 输出敏感信息检测告警：对模型输出进行 DLP 扫描，设定阈值告警

实践策略：当差分隐私难以满足业务性能要求时，可考虑组合措施——训练数据脱敏、严格访问控制、输出过滤。这并非完美方案，仍存在残余风险，但在隐私与性能之间取得可接受的平衡。

## 当前威胁态势

AI 安全威胁已从学术研究转向实际攻击。需要重点关注的威胁类型：

| 威胁类型 | 攻击方式 | 影响 | 紧迫度 |
|---------|---------|-----|-------|
| **Prompt Injection** | 直接注入（用户输入中包含恶意指令）或间接注入（通过外部数据源注入） | 绕过系统指令、安全护栏 | P0 |
| **供应链投毒** | 分发恶意模型或预训练权重 | 模型包含后门或恶意行为 | P0 |
| **LLM 幻觉滥用** | 利用 LLM 生成虚假但可信的内容 | 虚假信息、钓鱼攻击 | P1 |
| **模型窃取** | 大量 API 查询推断模型参数或蒸馏知识 | 知识产权损失 | P1 |

![LLM 安全风险全景](../../../assets/images/chapter_15/01_LLM_Security_Risks_v6.png)

优先级说明：当前最紧迫的威胁是 Prompt Injection。原因：
- 多数 LLM 应用在上线时缺乏针对性防护
- 攻击门槛低（无需专业 ML 知识）
- 影响范围广（所有 LLM 应用均受影响）
- 已有公开的攻击案例和工具

## 业务影响与投资决策

### 事件代价

AI 安全事件的代价包括多个维度：

| 损失类型 | 具体项目 | 量化参考 |
|---------|---------|---------|
| 直接损失 | 监管罚款、赔偿、法律费用 | EU AI Act 最高罚款可达年收入 7%（AI 特定违规） |
| 间接损失 | 声誉损害、业务中断、知识产权损失、信任危机 | 难以量化，但往往超过直接损失 |
| 恢复成本 | 事件响应、系统加固、客户沟通 | 取决于事件规模 |

关键约束：AI 安全事件的声誉损害往往超过直接经济损失。合规罚款虽有明确上限，但信任丧失的恢复成本难以估算。

### 投资决策框架

AI 安全投资需分阶段规划，根据组织 AI 采用阶段调整投入：

| 阶段 | 预算占比参考 | 人力配置 | 重点领域 |
|-----|-------------|---------|---------|
| 初始阶段（AI 刚起步） | AI 总预算的 5-10% | 安全工程师兼职 | 基础防护（访问控制、数据加密、输入验证） |
| 成长阶段（AI 规模化应用） | AI 总预算的 10-15% | ModelSec 专职团队（2-3 人） | OWASP LLM Top 10 前三项、供应链安全、红队测试 |
| 成熟阶段（AI 成为业务关键） | AI 总预算的 15-20% | ModelSec 团队扩充（5+ 人） | 合规认证、高级防御、持续红队验证 |

预算占比为参考值，需根据行业特性、合规要求和风险偏好调整。

常见误区：

| 误区 | 事实 | 正确做法 |
|-----|-----|---------|
| 一次性投入思维 | AI 安全是持续过程，威胁演进快 | 规划年度持续投入预算 |
| 仅关注技术投入 | 忽视组织能力建设、流程建设、人员培训 | 技术、流程、人员三位一体投入 |

验证方法：
- 定期评估 AI 安全预算占 AI 总预算的比例是否合理
- 对标同行业企业投入水平（注意行业差异）
- 跟踪安全投入与风险事件发生率的关系

### 为何需要专门的 AI 安全能力

AI 安全与传统安全存在显著差异，现有安全团队难以直接复用。下表从四个维度对比两者差异，选择这些维度的依据是：威胁类型决定防御策略，防御范式决定技术路线，工具决定实施手段，专业知识决定团队能力要求。

| 维度     | 传统安全           | AI 安全                      |
| -------- | ------------------ | ---------------------------- |
| 威胁类型 | 代码漏洞、配置错误 | 对抗样本、数据投毒、模型窃取 |
| 防御范式 | 补丁、访问控制     | 对抗训练、差分隐私、输入验证 |
| 工具     | SAST/DAST/IAST     | ModelScan、Garak、ART        |
| 专业知识 | 应用安全、网络安全 | ML 原理、统计学、优化理论    |

上表表明，AI 安全需要补充而非替代现有安全团队。现有团队负责基础设施安全，新建 ModelSec 团队专注 AI 特有威胁，两者协作共同覆盖完整威胁面。

## 常见误区与教训

下表归纳 AI 安全实践中的典型误区。选择这些误区的依据是：它们在企业实践中反复出现，且后果严重。

| 误区 | 后果 | 教训 | 识别信号 |
|-----|-----|-----|---------|
| AI 安全等于传统安全 | Prompt injection 零防护，数据泄露 | AI 威胁需要 AI 特有的防御手段 | 未配置 LLM 专用防护 |
| 完全信任第三方模型 | 模型可能存在后门或恶意行为 | 必须进行模型扫描、沙箱测试、溯源验证 | 直接部署未审核的开源模型 |
| 差分隐私可解决一切 | 准确率下降导致业务拒绝 | 隐私与性能权衡困难，需要组合措施 | 仅依赖差分隐私而忽视其他措施 |
| 对抗训练是银弹 | 准确率下降，训练成本增加 | 需评估成本收益，可能需要替代策略 | 未评估对抗训练对业务指标的影响 |
| 合规即安全 | 通过认证后仍被攻击 | 合规是基线，不是终点 | 以合规审计代替安全测试 |
| AI 伦理只是合规要求 | 偏见/公平性问题演变为法律和声誉危机 | 伦理问题需主动治理 | 未进行公平性评估 |

核心教训：AI 安全没有银弹。每个防御措施都有代价，需要在安全、性能、成本、用户体验之间持续权衡。坦诚技术边界和残余风险，是建立信任的基础。

## 成功衡量标准

AI 安全的成功不是"零事件"（这不现实），而是风险可控、业务可持续。

### 指标体系

| 指标类型 | 具体指标 | 参考目标 | 衡量方法 |
|---------|---------|---------|---------|
| **领先指标（预防性）**  | | | |
| | ModelSec 团队或专职角色建立 | 成长阶段完成 | 组织架构审查 |
| | 高风险 AI 系统安全评审覆盖率 | 100% | 评审记录 |
| | LLM 应用 Prompt Injection 防护覆盖率 | >95% | 部署审计 |
| | 第三方模型扫描验证率 | 100% | 模型仓库记录 |
| | 红队测试执行频率 | 每季度 | 测试报告 |
| **滞后指标（检测性）**  | | | |
| | AI 安全事件趋势 | 同比下降 | 事件统计 |
| | 红队发现高危漏洞修复率 | >90% | 漏洞跟踪 |
| | 合规审计发现项关闭率 | >95% | 审计报告 |
| **业务成果** | | | |
| | 对抗攻击成功率 | <10%（与无防护对比） | 红队测试 |
| | 合规认证 | 按时完成 | 认证证书 |
| | 客户 AI 安全评估 | 通过率 >90% | 客户反馈 |

## 本节小结

| 要点 | 说明 |
|-----|-----|
| AI 安全与传统安全本质不同 | 对抗样本、数据投毒、模型记忆泄露是 AI 特有威胁，需要 AI 特有防御 |
| 技术边界需坦诚 | 对抗样本在数学上难以根除，只能提高攻击成本 |
| 权衡贯穿始终 | 差分隐私与准确率、对抗训练与性能、安全与用户体验之间不存在银弹 |
| 投资需分阶段 | 根据 AI 采用阶段调整投入重点和规模 |
| 团队需协作 | 现有安全团队与 ModelSec 团队各有分工，协作配合 |
